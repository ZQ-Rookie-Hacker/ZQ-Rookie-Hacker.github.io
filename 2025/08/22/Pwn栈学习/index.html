1.内存布局

gdb中执行vmmap (以x86为例->对应上图中第一个表)

  • /proc/[pid]/maps：是Linux内核通过proc文件系统对外暴露的进程内存映射信息，是内核直接提供的“原始数据”，任何时候查看都能反映进程当前的内存状态（除非进程正在动态修改内存映射，可能存在极短的时间差）。
  • GDB的vmmap命令：其数据来源正是/proc/[pid]/maps（或通过ptrace系统调用从内核获取相同信息），只是GDB会对数据进行格式化处理，以更符合调试场景的方式展示（比如标注内存区域的权限、用途、对应的文件路径等）。
对比项 cat /proc/[pid]/maps GDB的vmmap命令
数据格式 原始文本，每行对应一个内存区域，字段紧凑（地址范围、权限、偏移量、设备号、inode、路径） 格式化输出，增加可读性（如用颜色标注权限、明确标注“stack”“heap”“code”等区域类型）
使用场景 适用于在终端直接查看、脚本解析（如批量分析进程内存） 仅用于GDB调试时，配合断点、内存查看等调试操作，方便定位漏洞或分析内存布局
实时性 实时反映进程当前状态（内核直接输出） 需在GDB中重新执行vmmap才能刷新（若进程在调试中修改了内存映射，旧输出会过时）
附加信息 仅包含内核记录的基础信息 可能结合调试符号（如ELF文件的section信息），更详细地标注代码段、数据段等
  • 代码段（Text Segment）：存放程序的机器码（二进制指令），通常位于最低的地址区域，具有只读和可执行权限。
  • 数据段（Data Segment）：存放已初始化的全局变量和静态变量。
  • BSS段：存放未初始化的全局变量和静态变量，运行时会被初始化为0。
  • 堆（Heap）：动态内存分配区域（如malloc/new），从低地址向高地址增长。堆的范围很大
  • 栈（Stack）：存放函数调用栈帧、局部变量、返回地址等，从高地址向低地址增长。

为什么代码在低地址，动态数据在高地址？

  • 历史原因：早期的系统（如x86 32位）地址空间有限，代码段通常被加载到固定的低地址（如0x08048000），而栈则被分配在高地址（如0xC0000000以下）。
  • 安全考虑：现代系统通过**地址空间布局随机化（ASLR）**增强安全性，将代码段、堆、栈等区域的起始地址随机化，但总体趋势仍是代码在低地址，栈在高地址。
  • 内存增长方向：
    • 堆：向高地址扩展（通过sbrk/mmap系统调用）。
    • 栈：向低地址扩展（每次函数调用时压栈）。

2.栈布局及其关联的寄存器

  • ebp（Extended Base Pointer）：x86 32位架构中的基址指针寄存器，指向当前栈帧（Stack Frame）的底部，即栈底指针。
  • rbp（Base Pointer）：x86_64 64位架构中的基址指针寄存器，功能与 ebp 相同，但扩展为64位。
  • esp（Extended Stack Pointer）：x86 32位架构中的栈指针寄存器，指向当前栈的顶部（即最新压入的数据位置）,即栈顶指针。
  • rsp（Stack Pointer）：x86_64 64位架构中的栈指针寄存器，功能与 esp 相同，但扩展为64位。

简单来说：ebp/rbp 是基址指针，esp/rsp 是栈指针，分别用于定位栈帧的底部和顶部。

与栈的关系：

在x86/x86_64架构中，栈是向下生长的（即从高地址向低地址扩展）。每次函数调用时，系统会创建一个新的栈帧（Stack Frame），用于存储：

  1. 返回地址：调用函数后应返回的指令位置。
  2. 调用者的 ebp 值：用于恢复上一个栈帧。
  3. 局部变量：函数内部定义的变量。
  4. 参数：传递给被调用函数的参数。

ebp/rbp 和 esp/rsp 的具体作用：

  • ebp/rbp：固定指向当前栈帧的底部（即存储上一个栈帧的 ebp 值的位置），用于访问局部变量和参数（通过偏移量计算）。
  • esp/rsp：动态指向栈的顶部，随着数据的压栈（PUSH）和出栈（POP）操作而变化。

64位与32位的差异

  • 寄存器大小：64位架构使用64位的 rbp 和 rsp，而32位使用32位的 ebp 和 esp。
  • 参数传递：x86_64中，前6个整数/指针参数通过寄存器（rdi, rsi, rdx, rcx, r8, r9）传递，而不是全部压栈，因此栈帧结构可能更简洁。

函数调用约定

（1） __cdecl
  • 简介：这是C/C++语言默认的函数调用约定。
  • 特点：参数从右向左入栈，主调函数负责栈平衡。在C/C++中，当函数参数数量不确定（如使用可变参数函数，像printf）时，通常使用这种调用约定，因为主调函数在知道实际参数数量后才能正确清理栈空间。
（2） __stdcall
  • 简介：这是Windows API默认的函数调用方式。
  • 特点：参数从右向左入栈，被调函数负责栈平衡。在Windows系统编程中，系统提供的API大多采用这种调用约定，这样可以使函数调用的栈操作由被调用函数自身处理，在一定程度上简化主调函数的实现。
（3） __fastcall
  • 简介：这是一种快速调用方式。
  • 特点：参数优先从寄存器（ECX和EDX）传入，剩下的参数再从右向左从栈传入。因为栈位于内存区域，而寄存器位于CPU内，寄存器的存取速度比内存快很多，所以这种方式能提高函数调用的效率，适用于对性能要求较高的场景。

阶段1：进入main函数，执行push ebp（main <+0>）

  • 操作：push ebp（保存调用者的ebp到栈）
  • 栈变化：esp从初始0x1000减4 → 0xFFC，栈顶存储旧ebp（假设调用者ebp为0x2000）
  • 栈状态：
  • esp → 0xFFC
  • ebp → 0x2000（未更新）
  • eip → 0x565561b1（下一条指令地址）

阶段2：执行mov ebp, esp（main <+1>）

  • 操作：mov ebp, esp（设置main函数的ebp为当前栈顶）
  • 栈状态不变，仅ebp更新：
  • esp → 0xFFC
  • ebp → 0xFFC（与esp相同）
  • eip → 0x565561b3（下一条指令地址）

阶段3：执行sub esp, 0x10（main <+3>）

  • 操作：sub esp, 0x10（为main的局部变量分配0x10字节栈空间）
  • 栈变化：esp从0xFFC减0x10 → 0xFEC
  • 栈状态（0xFEC到0xFFB为局部变量空间）：
  • esp → 0xFEC
  • ebp → 0xFFC
  • eip → 0x565561b6（下一条指令地址）

阶段4：main函数初始化局部变量（<+16>和<+23>）

  • 操作：
    • mov DWORD PTR [ebp-0x8], 0x14 → 局部变量1（0x14=20）存在ebp-0x8（0xFF4）
    • mov DWORD PTR [ebp-0x4], 0x1e → 局部变量2（0x1e=30）存在ebp-0x4（0xFF8）
  • 栈状态：
  • esp → 0xFEC
  • ebp → 0xFFC
  • eip → 0x565561ce（下一条指令地址）

阶段5：调用add_test前压入参数（<+36>到<+42>）

  • 操作（C调用约定：从右到左压参）：
    • push DWORD PTR [ebp-0x4] → 压入参数3（30），esp=0xFEC-4=0xFE8
    • push DWORD PTR [ebp-0x8] → 压入参数2（20），esp=0xFE8-4=0xFE4
    • push eax → 压入参数1（假设eax=0x56558ffc，字符串地址），esp=0xFE4-4=0xFE0
  • 栈状态（参数从高到低：参数3、参数2、参数1）：
  • esp → 0xFE0
  • ebp → 0xFFC
  • eip → 0x565561db（下一条指令地址）

阶段6：执行call add_test（main <+43>）

  • 操作：call add_test（压入返回地址，跳转到add_test）
  • 栈变化：压入返回地址（main <+48>的地址0x565561e0），esp=0xFE0-4=0xFDC
  • 栈状态：
  • esp → 0xFDC
  • ebp → 0xFFC
  • eip → 0x5655617d（add_test入口地址）

阶段7：进入add_test，执行push ebp（<+0>）

  • 操作：push ebp（保存main的ebp到栈）
  • 栈变化：esp=0xFDC-4=0xFD8，栈顶存储main的ebp（0xFFC）
  • 栈状态：
  • esp → 0xFD8
  • ebp → 0xFFC（未更新）
  • eip → 0x5655617e（下一条指令地址）

阶段8：执行mov ebp, esp（add_test <+1>）

  • 操作：mov ebp, esp（设置add_test的ebp为当前栈顶）
  • 栈状态不变，仅ebp更新：
  • esp → 0xFD8
  • ebp → 0xFD8（与esp相同）
  • eip → 0x56556180（下一条指令地址）

阶段9：执行sub esp, 0x10（add_test <+3>）

  • 操作：sub esp, 0x10（为add_test的局部变量分配0x10字节栈空间）
  • 栈变化：esp=0xFD8-0x10=0xFC8
  • 栈状态（0xFC8到0xFD7为局部变量空间）：
  • esp → 0xFC8
  • ebp → 0xFD8
  • eip → 0x56556183（下一条指令地址）

阶段10：add_test初始化局部变量（<+16>）

  • 操作：mov DWORD PTR [ebp-0x4], 0x4 → 局部变量（0x4=4）存在ebp-0x4（0xFD4）
  • 栈状态：
  • esp → 0xFC8
  • ebp → 0xFD8
  • eip → 0x56556194（下一条指令地址）

阶段11：add_test执行完运算，执行leave（<+49>）

  • 操作：leave（等价于mov esp, ebp; pop ebp）
    • 第一步：mov esp, ebp → esp=0xFD8
    • 第二步：pop ebp → 弹出main的ebp（0xFFC）到ebp，esp=0xFDC
  • 栈状态（局部变量空间被释放）：
  • esp → 0xFDC
  • ebp → 0xFFC
  • eip → 0x565561af（下一条指令地址）

阶段12：add_test执行ret（<+50>）

  • 操作：ret（弹出返回地址到eip，栈顶指向参数）
  • 栈变化：弹出返回地址（0x565561e0）到eip，esp=0xFDC+4=0xFE0
  • 栈状态：
  • esp → 0xFE0
  • ebp → 0xFFC
  • eip → 0x565561e0（main <+48>地址）

阶段13：main清理栈参数（<+48>）

  • 操作：add esp, 0xc（释放3个参数的栈空间，0xc=3×4）
  • 栈变化：esp=0xFE0+0xc=0xFEC
  • 栈状态（参数空间被释放，回到main的局部变量）：
  • esp → 0xFEC
  • ebp → 0xFFC
  • eip → 0x565561e3（下一条指令地址）

阶段14：main执行leave（<+56>）

  • 操作：leave（等价于mov esp, ebp; pop ebp）
    • 第一步：mov esp, ebp → esp=0xFFC
    • 第二步：pop ebp → 弹出旧ebp（0x2000）到ebp，esp=0x1000
  • 栈状态（main的局部变量空间被释放）：
  • esp → 0x1000
  • ebp → 0x2000（恢复为调用者的ebp）
  • eip → 0x565561e9（下一条指令地址）

阶段15：main执行ret（<+57>）

  • 操作：ret（弹出调用者的返回地址到eip，程序结束）
  • 栈变化：esp=0x1000+4=0x1004
  • 最终状态：程序返回到调用main的函数
  • esp → 0x1004
  • ebp → 0x2000
  • eip → 调用者的返回地址

（10）PLT与GOT

PLT（Procedure Linkage Table，过程链接表）和GOT（Global Offset Table，全局偏移表）是动态链接机制中用于实现外部函数调用的关键数据结构，主要解决程序运行时对动态库中函数地址的动态绑定问题。

基本定义

  • PLT：由一系列汇编代码片段组成的表结构，每个外部函数对应一个PLT条目。其核心功能是引导函数调用流程，在首次调用时触发地址解析，后续调用则直接跳转至已解析的地址。
  • GOT：由一系列地址项组成的表结构，与PLT条目一一对应。用于存储外部函数的真实内存地址，首次调用前未初始化，首次调用后被动态链接器更新为目标函数的实际地址。

核心作用
在程序调用动态库中的外部函数（如printf）时，由于编译阶段无法确定这些函数在内存中的最终地址（动态库加载地址不固定），PLT和GOT通过协同工作实现地址的动态解析与复用，具体流程如下：

首次调用外部函数

  1. 程序触发调用：执行call指令时，实际跳转至该函数对应的PLT条目。
  2. PLT引导至GOT：PLT条目中的指令首先尝试跳转到GOT中对应的地址项。此时GOT项尚未初始化，其值指向该PLT条目内的后续指令（形成“回跳”）。
  3. 启动地址解析：回跳后，PLT条目通过压入函数的重定位索引等信息，跳转至PLT的第0个通用条目，进而触发动态链接器的解析函数（如_dl_runtime_resolve）。
  4. 动态链接器更新GOT：解析函数根据索引从动态库中找到目标函数的真实地址，并将该地址写入GOT中对应的条目。
  5. 执行目标函数：解析完成后，直接跳转至目标函数的真实地址执行。

解释2，3： 开始是函数对应的got表项为空，即没有其函数核心代码的地址，而此时的函数核心代码的地址在函数对应的plt表项的后半部分。则过程是，执行到plt表项前部分，跳转至got,发现got为空,然后plt前部分跳到plt后部分，然后plt后部分跳转至plt前部分更前面，经过一些跳转，会将got处写上函数核心代码的地址

非首次调用外部函数

  1. 程序触发调用：同样跳转至对应PLT条目。
  2. 直接使用GOT中的地址：PLT条目指令跳转到GOT对应项时，该地址项已被更新为目标函数的真实地址，因此直接跳转至目标函数执行，无需再次解析。

解释：由于此时函数对应的got表项存储函数核心代码的地址，函数对应的plt表项的前部分跳转到got表项并利用其存储的地址跳转至函数核心代码处，所以函数对应的plt表项的后半部分不再被程序执行

总结
PLT与GOT的配合实现了“延迟绑定”机制：仅在函数首次被调用时进行地址解析并更新GOT，后续调用直接复用GOT中存储的真实地址。这一机制减少了程序启动时的地址解析开销，提升了动态链接的效率。

注意： 这是在保护机制为 partial relocation read only. 若是 full relocation read only 则一开始函数核心代码地址就在got中

1.什么是libc?

libc 即 GNU C 标准库（GNU C Library） ， 封装了大量常用的函数，为程序员提供了一系列基础且关键的功能，涵盖了输入输出操作、内存管理、字符串处理、数学运算、进程控制等多个方面。
比如，常见的 printf 函数用于格式化输出，malloc 和 free 函数分别用于动态内存的分配与释放 等等

组成结构:

头文件：定义了库中函数、类型和宏的声明，像 <stdio.h> 声明了输入输出相关函数（如 printf、scanf 等），<stdlib.h> 声明了内存管理（malloc、free）、进程终止（exit）等函数，<string.h> 声明了字符串操作函数（strcpy、strlen 等）。程序员在编写代码时，通过 #include 指令引入这些头文件，就能使用对应的函数。
目标文件：经过编译和链接后，形成了包含实际函数代码的二进制文件。在 Linux 系统中，动态链接版本的 libc 通常是 libc.so.6 ，程序在运行时，动态链接器会将这个共享库加载到内存中，让程序能够调用其中的函数。

（8）栈迁移

栈迁移是一种在缓冲区溢出攻击中常用的高级技术，其本质是通过控制栈指针（如 x86 的ESP、x64 的RSP）和伪造栈帧结构，将程序当前的 “活跃栈帧” 从原始栈空间迁移到其他可控内存区域（如堆、数据段、共享库可读可写区等），从而绕过防护或扩展攻击空间。

问题场景 传统方法限制 栈迁移解决方案
栈溢出空间不足 ROP链无法完整布置 迁移到大容量内存区域
NX防护（栈不可执行） 无法直接执行shellcode 迁移到可执行内存区域
只能覆盖有限数据 无法控制完整控制流 只需覆盖关键指针即可
ASLR随机化 难以预测绝对地址 使用相对稳定的堆地址

其主要目的是：将栈迁移到 bss 段或者是空间足够大的 heap 上或者是 libc 上

函数退出过程

步骤 1：函数退出前的初始栈状态
函数执行到 return 前，栈中包含局部变量、保存的旧 ebp、返回地址及上层栈帧内容。

**步骤 2：**执行 leave 指令后（等价于 mov esp, ebp; pop ebp）
leave 指令分两步修改栈和寄存器：

  1. mov esp, ebp：将栈顶指针 esp 直接设置为基址指针 ebp 的值（即 esp = ebp = 0xFFC）。
  2. pop ebp：从 esp 指向的地址（0xFFC）弹出值（更上层的旧 ebp，值为 0x0）到 ebp，然后 esp 自增 4（因 x86 栈按 4 字节对齐，弹出后栈顶上移）。

步骤 3：执行 ret 指令后（等价于 pop eip）
ret 指令的作用是从栈顶弹出返回地址到指令指针 eip，然后 esp 再自增 4（弹出 4 字节地址后，栈顶继续上移）。

引入：

.data：存储已初始化的全局 / 静态变量，有读写（rw）权限

.bss ：存储未初始化的全局 / 静态变量，同样有读写（rw）权限（运行时会被系统初始化为 0）

在程序的符号表和代码结构中，这两处与 system 相关的部分有着不同的角色和作用

第一处（extern:0804A038 相关）

  • 这是对 system 函数的外部符号声明。在程序中，system 是一个来自标准库（如 libc）的函数，这里的 extern 表示该函数的定义在当前程序外部（即库文件中）。
  • DATA XREF: .got.plt:off_804A010↑o 表明，这个外部符号与全局偏移表（GOT，Global Offset Table）中的某个条目相关联。在程序运行时，动态链接器会将 system 函数的实际内存地址填充到 GOT 表对应的位置，使得程序能够正确调用 system 函数。

第二处（.plt:08048320 相关）

  • 这是 system 函数的**过程链接表（PLT，Procedure Linkage Table）**条目。PLT 是用于处理动态链接函数调用的一种机制。
  • 当程序中第一次调用 system 函数时，会通过 PLT 条目（这里的 jmp ds:off_804A010）间接跳转到 GOT 表中存储的 system 函数地址（而 GOT 表中的地址在程序启动或首次调用时会被动态链接器解析为 system 函数在内存中的实际地址）。后续调用 system 函数时，就可以直接通过 GOT 表中的地址快速跳转，无需再次进行复杂的动态链接解析过程。


  1. elf.symbols['system']返回PLT地址的原因
    在编译为动态链接的程序中，system是libc库中的函数，程序本身不包含其实现代码，只通过PLT条目和GOT条目间接调用：
  • elf.symbols['system']获取的是程序中system函数的PLT入口地址（即.plt:08048320），这是程序内部用于调用system的“跳板地址”。
  • 当程序通过PLT地址（0x8048320）调用system时，会自动通过GOT表找到system在内存中的实际地址（0xf7db5220）并执行，这是动态链接的正常流程。
  1. 为什么PLT地址（0x8048320）也能getshell？
    虽然PLT地址不是system函数的实际代码地址，但它的作用就是间接跳转到实际的system函数：
  • 当你在漏洞利用中用PLT地址（0x8048320）覆盖返回地址时，程序执行到该地址后，会通过PLT->GOT的机制自动找到并执行system的实际代码（0xf7db5220）。
  • 只要system函数的参数（如/bin/sh的地址）正确，无论使用PLT地址还是实际地址，最终都会调用system("/bin/sh")，从而获取shell。
  1. 两者的区别与适用场景

  • PLT地址（0x8048320）：

    • 是程序自身固定的地址（因为No PIE），不随libc加载位置变化而变化，稳定性高。
    • 适用于不知道system实际地址（如未泄露libc时）的场景。

  • 实际地址（0xf7db5220）：

    • 是system函数在内存中的真实执行地址，由libc加载时决定，可能随系统/环境变化。
    • 适用于已泄露libc基址，能准确计算出system地址的场景。

ciscn_2019_n_8中的细节


在 IDA 的函数列表里，两个与 system 相关的函数主要区别如下：

  1. _system
    它一般是程序内部对 system 函数的封装或重命名。可能是为了适配程序自身的逻辑、调用约定，或是经过编译器、链接器处理后的符号。本质上，它最终会调用系统的 system 函数，但属于程序内部的一个中间过渡符号，相当于程序自己“转手”调用标准 system 函数的一个环节。

  2. 下方的 system
    这个 system 是外部符号（external），更直接关联标准 C 库（libc）里的 system 函数本身。因为 system 函数的实现是在 libc 库中，并非程序自身代码所写，所以反汇编工具会将其标记为 external，表示该函数的实现不在当前程序内部，需要通过动态链接（借助 PLT 过程链接表、GOT 全局偏移表机制）从系统库中加载后调用。

切片语法 截取范围 通俗理解
[-4:] 从倒数第4个到末尾 “拿最后4个”
[:-4] 从开头到倒数第4个之前（不含） “拿前面所有，除了最后4个”

通过具体例子能更直观区分：对于 s = "1234567890"（长度10），s[-4:] 是 '7890'，s[:-4] 是 '123456'。
