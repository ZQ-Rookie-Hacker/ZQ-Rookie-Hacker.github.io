浏览器同源策略与CORS解析

字数: 计算中... 阅读时长: 计算中...

前言：为什么需要同源策略？

互联网诞生之初，浏览器允许任意网页随意访问其他网站的数据。这带来了巨大的安全隐患：恶意网站可以悄悄读取你在银行网站上的个人信息，甚至冒充你进行操作。为了解决这个问题，浏览器引入了 同源策略，作为一道基本的安全防线。

但同源策略过于严格，现代 Web 应用又需要合法的跨域通信（比如前端调用不同域名的 API）。于是，W3C 制定了 CORS，在保证安全的前提下，为跨域访问打开一扇可控的门。

本笔记将按照 “问题 → 解决方案 → 细节机制 → 安全风险” 的逻辑，为你彻底讲清同源策略与 CORS。

1. 同源策略：浏览器安全的基石

问题由来

如果没有同源策略，当你登录银行网站 https://bank.com 后，再打开一个恶意网站 https://evil.com，恶意网站可以这样攻击：

由于浏览器会携带银行网站的 Cookie，请求会成功返回你的账户数据，然后被恶意网站窃取。

解决方案：同源策略

浏览器规定：只有协议、域名、端口完全相同的两个页面，才能通过脚本互相读取数据。不同源的页面发起的请求，浏览器会发送请求，但拦截响应，不让 JavaScript 拿到数据。这样就阻止了恶意网站读取银行的数据。

同源判断规则

对比 URL 判断步骤 结果
https://api.hello.com/forget 域名不同 ❌ 跨域
http://hello.com/api/login 协议不同 ❌ 跨域
https://api.hello.com:8080/register 端口不同 ❌ 跨域
https://hello.com/login 全部相同 ✅ 同源

同源策略限制了什么？

限制类型 说明 为什么这样限制
DOM 访问 无法读取跨域 iframe 的内容 防止恶意页面操作其他网站的界面
存储访问 无法读取跨域的 Cookie、LocalStorage 防止窃取登录凭证、本地数据
网络请求 跨域请求的响应被拦截 防止窃取敏感数据

例外：跨域写操作和嵌入资源

  • 表单提交（<form action="...">）不受限，因为只是写入（提交数据），不读取数据。
  • 图片、脚本、样式等资源 默认允许嵌入，因为这些资源是展示型的，脚本无法直接读取其内容（比如 <img> 虽然能加载图片，但无法用 JS 读取图片的原始像素数据）。

总结：同源策略是浏览器的“默认隔离”，让不同源的网页无法互相读取数据，从而保护用户隐私和安全。

2. CORS 的诞生：在安全与功能之间找平衡

问题由来

随着 Web 应用复杂化，前端代码经常需要调用不同源的 API（例如 web.example.com 调用 api.example.com）。如果完全禁止跨域读取，现代 Web 开发将寸步难行。我们需要一种机制，让服务器可以 主动声明 哪些源可以访问它的资源，浏览器根据声明决定是否放行。

解决方案：CORS（跨域资源共享）

CORS 通过在 HTTP 头部增加一系列 Access-Control-* 字段，让服务器告知浏览器：

  • 允许哪些源访问
  • 允许哪些方法、哪些头部
  • 是否允许携带凭证（Cookie、Authorization）
  • 预检请求的缓存时间

浏览器在收到响应后，检查这些头部，只有符合规则时才将数据交给 JavaScript。

CORS 的核心思想

  • 默认拒绝，显式授权：任何跨域请求，服务器必须明确返回允许的源，浏览器才会放行。
  • 预检机制：对于可能修改数据的请求（如 PUT、DELETE），先询问服务器是否允许，避免直接执行危险操作。

3. 简单请求：最简单的跨域场景

问题由来

对于简单的 GET、POST 请求（例如获取公开数据），我们希望尽量减少开销，直接发送一次请求即可。

解决方案：简单请求

浏览器定义了一类 简单请求（满足条件的方法、头、Content-Type），这类请求不触发预检，直接发送，但服务器仍然必须在响应中包含 Access-Control-Allow-Origin。

条件

  • 方法：GET、HEAD、POST
  • 只能使用简单头部（Accept、Accept-Language、Content-Language、Content-Type，且 Content-Type 只能是 application/x-www-form-urlencoded、multipart/form-data、text/plain）
  • 不能有自定义头部（如 X-Custom-Header）

交互步骤

为什么这样设计？

  • 简单请求 被认为是安全的，不会修改服务器资源，所以可以直接发送。
  • 但即便如此，服务器仍需明确授权，否则浏览器不会交出数据。

4. 非简单请求与预检：保护服务器的安全

问题由来

如果跨域请求可能修改服务器数据（如 PUT、DELETE），或携带自定义头（如 X-Auth-Token），服务器需要预先知道并决定是否允许。否则，恶意网站可能通过一次跨域请求直接删除数据。

解决方案：预检请求（OPTIONS）

浏览器在发送真实请求前，先发送一个 OPTIONS 请求，询问服务器：

  • 是否允许该 Origin？
  • 是否允许该方法（如 PUT）？
  • 是否允许携带这些自定义头？

服务器返回允许的配置，浏览器核对后，若允许才发送真实请求。这样，即使恶意网站发起跨域请求，如果服务器未授权，真实请求根本不会发出。

交互步骤

为什么需要预检？

  • 防止 CSRF 攻击：如果没有预检，恶意网站可以直接用 PUT 删除用户数据，而浏览器无法事先阻止。
  • 节省服务器资源：服务器在预检阶段就拒绝不满足条件的请求，避免后续处理。
  • 灵活授权：服务器可以针对不同的方法、头部精细控制。

5. 预检缓存：优化性能

问题由来

对于同一个 API 的多次非简单请求，每次都发送 OPTIONS 会造成不必要的网络开销，降低性能。

解决方案：预检缓存

服务器可以在预检响应中返回 Access-Control-Max-Age，告诉浏览器在指定秒数内，对于相同的请求（相同 URL、方法、头部）不再发送预检，直接发送实际请求。

交互步骤

为什么这样设计？

  • 减少网络往返，提升性能。
  • 缓存时间不宜过长，否则当服务器策略变更时，浏览器仍用旧策略，可能造成安全问题。

6. 携带凭证（Cookie/Authorization）的请求

问题由来

很多 API 需要验证用户身份，通常通过 Cookie 或 Authorization 头传递凭证。但默认情况下，跨域请求不会携带这些凭证，因为如果恶意网站能携带用户的 Cookie 跨域请求，就可能冒充用户。

解决方案：显式授权

  • 前端：必须在 fetch 或 XMLHttpRequest 中设置 credentials: 'include'（或 withCredentials = true），表示愿意发送凭证。
  • 服务器：必须在响应中返回 Access-Control-Allow-Credentials: true，且 Access-Control-Allow-Origin 不能为 *，必须指定具体源。
  • 浏览器只有同时满足这两个条件，才会在请求中带上 Cookie，并将响应中的 Set-Cookie 存入。

交互步骤

为什么这样设计？

  • 防止恶意网站未经用户同意就携带用户的凭证跨域请求（CSRF 攻击）。
  • 要求前端显式启用凭证，并要求服务器明确授权，双重保障。

7. 漏洞场景 ①：反射 Origin 未严格校验

问题由来

有些服务器为了方便，直接将请求的 Origin 原样反射到 Access-Control-Allow-Origin 中，而没有校验该 Origin 是否在白名单内。攻击者可以构造一个恶意网站，诱使用户访问，然后向该服务器发起请求。服务器反射了攻击者的 Origin，浏览器认为被授权，将数据交给恶意网站。

攻击流程

为什么会出现这种漏洞？

  • 服务器错误地认为只要反射 Origin 就能实现跨域，忽略了白名单校验。
  • 这种配置常见于开发环境或快速原型中，被误带到生产环境。

防御

  • 服务器必须维护一个严格的白名单，对每个请求的 Origin 进行完全匹配校验。
  • 只有白名单中的源才返回 Access-Control-Allow-Origin。

8. 漏洞场景 ②：内部网络暴露

问题由来

许多企业内部服务（如管理后台、监控面板）部署在内网，且为了方便，配置了 Access-Control-Allow-Origin: *。当用户通过浏览器访问外网恶意网站时，恶意网站可以诱导用户的浏览器向内网服务发起跨域请求，由于内网服务允许任意源，浏览器会放行，攻击者就能读取内网敏感数据。

攻击流程

为什么会出现这种漏洞？

  • 内网服务开发者认为“内部使用，不需要 CORS 限制”，却忽略了浏览器跨域请求可能来自外部网页。
  • 即使服务部署在内网，只要用户能访问该服务，攻击者就能利用浏览器作为代理。

防御

  • 内网服务不应配置 Access-Control-Allow-Origin: *。
  • 如果需要跨域，应限制到具体的信任源（如公司内部域名）。
  • 敏感服务建议仅允许同源访问，或使用更严格的认证（如 IP 白名单、VPN）。

9. 最佳实践：服务器端白名单校验

问题由来

如何安全地实现 CORS，避免上述漏洞？

解决方案：严格白名单校验

关键点

  • 白名单：应包含完整的协议、域名、端口（如 https://web.example.com），不要使用通配符。
  • 动态反射：如果采用动态反射，务必对 Origin 进行白名单匹配，且只允许白名单内的源。
  • 凭证请求：当需要凭证时，Allow-Origin 必须返回具体源，不能为 *。
  • 预检缓存：设置合理的 Max-Age（如 3600 秒），避免缓存时间过长导致策略更新不及时。

10. 总结：核心知识一览

编号 知识点 为什么存在？ 核心机制
1 同源策略 防止恶意网站窃取跨域数据 只有同源的页面才能互相读取数据
2 简单请求 对安全操作最小化开销 一次请求，服务器返回 Allow-Origin
3 非简单请求 + 预检 防止危险操作未经授权执行 先 OPTIONS 询问，再实际请求
4 预检缓存 减少 OPTIONS 请求，提升性能 Access-Control-Max-Age 控制缓存时间
5 凭证请求 允许携带 Cookie/Authorization，但需显式授权 credentials: 'include' + Allow-Credentials: true + 明确源
6 反射 Origin 漏洞 服务器未校验 Origin，导致任意源可获取数据 攻击者利用反射获得授权
7 内网暴露漏洞 内网服务允许所有源，攻击者可读取内网数据 浏览器作为跳板，窃取内网信息
8 防御最佳实践 避免 CORS 配置错误 严格白名单、最小权限、合理缓存
