0x00:为什么可以hook？

Hook 之所以能生效，是 JavaScript 语言特性与浏览器架构共同决定的。

浏览器提供了可拦截的 API 层

浏览器将底层能力（网络请求、DOM 操作、Cookie 读写等）通过 Web API 暴露给 JS。这些 API 在底层由 C++ 实现，但在 JS 层面，它们以对象属性和方法的形式挂在 window、document、XMLHttpRequest.prototype 等全局对象上。关键在于：浏览器没有对这些 JS 层的绑定做只读保护，所以我们可以在 JS 层替换它们。

同时，浏览器是单线程执行模型，我们的 hook 脚本和目标页面代码运行在同一个 JS 上下文中，共享同一个全局作用域和对象引用。这意味着你修改了 JSON.parse，目标代码调用 JSON.parse 时就会走进你注入的函数。

在此基础上，JS 的三个语言特性让 hook 变得简单：

1. 函数是第一类对象（First-class Object）

在 JS 中，函数和普通对象一样可以被赋值、传递、覆盖。浏览器内置的 API（如 document.cookie 的 setter、XMLHttpRequest.prototype.open）本质上也是挂载在对象上的属性，所以可以直接被替换：

这就是最基础的 hook —— 保存原引用，替换为新函数，新函数内部再调用原引用。

2. Object.defineProperty 可以重定义属性的 getter/setter

对于像 document.cookie 这类通过属性访问器（getter/setter）实现的属性，普通赋值无法拦截。但 Object.defineProperty 可以重新定义属性的 get 和 set，在读写时插入自定义逻辑：

这也是为什么文章中大量 hook 脚本都用 Object.defineProperty —— 它能拦截属性的读写行为，而不仅仅是函数调用。

3. 原型链机制（Prototype Chain）

JS 的方法查找沿原型链进行。修改原型上的方法，所有实例都会受到影响。例如 hook XMLHttpRequest.prototype.open，页面中所有 XHR 请求的 open 调用都会经过你注入的函数，无需逐个实例去改。

实际应用中的两种注入时机：

  • 油猴/注入脚本：在页面 JS 执行前注入（@run-at document-start），抢在目标代码之前完成 hook
  • DevTools Overrides：用本地文件覆盖线上 JS，在源码层面插入断点或 hook 代码

理解了这三点，下面所有 hook 脚本的套路都是一样的：保存原函数 → 替换为自定义函数 → 自定义函数中决定是否断下/打印/篡改 → 调用原函数保持正常行为。
