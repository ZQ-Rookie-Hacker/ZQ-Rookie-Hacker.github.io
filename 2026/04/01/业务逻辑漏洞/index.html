0x08.工作流程验证不足

  • 任务：买一件“轻便l33t皮夹克”

  • 提示：对采购流程中的事件顺序做出了错误的假设

  • 解释

    后端没有校验你有没有走完业务流程，允许你直接跳过所有中间步骤，强行请求最终的 “执行接口”，让操作直接生效。
    大部分网站的功能都是分步骤的业务流程，比如：

    1.购物：选商品 → 加入购物车 → 结算 → 支付 → 下单成功
    2.改密：验证旧密码 → 输入新密码 → 确认
    3.转账：输入金额 → 验证短信 → 确认转账

    正常逻辑：必须走完前面所有步骤，才能执行最后一步。而 Insufficient workflow validation 就是：
    后端不检查你是否完成了前置步骤，你直接请求最后一步的接口，操作照样生效

正常付款 -> 返回包存在重定向

0x09.通过有缺陷的状态机实现的认证绕过

  • 任务：访问管理员界面，删除用户 carlos
  • 提示：对登录过程中的事件顺序做出了错误的假设
  • 解释

状态机是应用程序中管理对象生命周期和行为流转的核心机制，就像自动售货机的工作流程：

1.初始状态：等待投币（未登录）
2.中间状态：已投币，等待选择商品（已登录但未选择角色）
3.最终状态：已出货，完成交易（已选择角色，正常访问系统）

在安全场景中，状态机缺陷指攻击者通过构造非预期的状态转移请求，绕过正常业务流程限制，进入未授权状态。

正常的登录流程：

1.用户提交账号密码（wiener:peter）
2.服务器验证通过，创建会话
3.强制跳转到 /role-selector 页面（用户需选择 “User” 或 “Content author” 角色）
4.用户选择角色后，服务器记录角色信息，允许访问系统

此环境的关键漏洞点:

1.用户必然会进入角色选择页面：没有在后端强制校验 “角色是否已选择”
2.未选择角色时会有默认安全值：实际未选择角色时，系统默认分配了最高权限的 administrator 角色，而非最低权限的 guest 角色

登录之后需要选择一个角色，说明后端不知道账户的角色

登录的时候就进行劫持，转到/admin

结果

原因也说过了，因为在不知道用户角色的时候默认是admin,不进入选择角色页面，直接重定向到/admin （此时的权限即admin权限是可以访问的）

0x0b.通过加密预言机绕过认证

  • 任务：删除该用户 carlos
  • 提示：存在一个逻辑缺陷，暴露给用户加密预言机
  • 解释

在加密点可以“制作”任意内容的密文，然后在解密点可以“读取”任意密文对应的明文。这就是所谓的“加密预言机”

一个例子解释：

  1. 正常登录，抓取关键 Cookie

    用提供的账号 wiener:peter 登录，记得勾选“Stay logged in”。
    登录成功后，在 Burp 的 Proxy → HTTP history 里，找到包含 stay-logged-in cookie 的响应包。这个 cookie 看起来像一串乱码，是加密后的 username:timestamp 格式。
    2. 触发加密点 (Encryptor)

    发表一条评论，在邮箱（Email）字段里随便输入一个无效的邮箱地址，比如 test。
    提交后会收到一个重定向，响应头里会设置一个新的 notification cookie。
    找到这个设置 notification cookie 的响应包，把它发送到 Repeater，重命名为 Encryptor。这个 POST 请求就是你的“加密机”——你修改邮箱参数，它能返回加密后的值（就是新的 notification cookie）。
    3. 触发解密点 (Decryptor)

    接着访问重定向后的页面，此时请求里会带着刚才那个 notification cookie。
    找到这个 GET 请求的响应包，里面会明文显示 Invalid email address: test 这样的错误信息。
    把这个 GET 请求也发送到 Repeater，重命名为 Decryptor。这个 GET 请求就是你的“解密机”——你修改 notification cookie 的值，它能在错误消息里显示对应的明文

思路总结

  1. 完整走一遍正常流程，摸清业务链路

    • 正常注册、登录、下单、支付、改密、查看资料等，走完所有步骤。
    • 抓包记录：接口、参数、顺序、关键字段（金额、数量、角色、状态、ID、邮箱、Cookie）。
    • 搞懂：系统期望你怎么操作，这是破坏逻辑的前提。

  2. 无视前端，只盯服务端校验

    • 前端限制（JS校验、按钮灰掉、输入框限制）全部默认可绕过。
    • 核心判断标准：服务端到底有没有校验、校验了什么。

  3. 寻找“逻辑断点”——只找这几类薄弱点

    • 关键参数是否可随意改：价格、数量、商品ID、权限标识、用户ID、订单状态。
    • 业务流程是否可跳步：不登录进后台、不加购直接支付、不验证直接改密。
    • 状态机是否混乱：登录后不选角色直接进管理页、订单状态可手动篡改。
    • 规则是否有漏洞：优惠券可重复用、积分可刷、金额可溢出。
    • 输入解析是否不一致：邮箱/手机号前后端解析不一样、超长截断、编码绕过。
    • 权限是否混用：同一个接口普通用户/管理员共用，仅靠参数区分权限。

  4. 统一试探手法：构造“非正常请求”

    • 改参数：把价格改成0、数量改成负数、用户ID改成管理员ID。
    • 跳步骤：直接请求最后一步接口，跳过前面验证。
    • 改状态：把订单状态“未支付”改成“已支付”。
    • 边界值：超大整数（溢出）、超长字符串（截断）、特殊字符、畸形编码。
    • 重复操作：重复提交、重复兑换礼品卡、重复使用优惠券。

  5. 单点不行就组合利用

    • 单个参数无效 → 搭配多个参数一起改。
    • 直接改金额失败 → 用负数+优惠券组合实现0元。
    • 无法直接越权 → 用加密预言机、邮箱绕过、Cookie伪造间接提权。

  6. 自动化放大漏洞效果

    • 重放、循环请求：刷钱、刷积分、无限兑换。
    • 批量篡改：批量越权、批量查询数据。
    • 用宏/脚本自动跑完整个利用链。

先摸清正常流程 → 找到服务端没校验的地方 → 篡改参数/跳步骤/改状态 → 组合利用达到越权、免费、刷资源等目的。
