0x02 小程序 AES 加密 id 参数越权——逆向反编译 + 批量遍历拿下十万用户敏感信息

在平台更新新资产后，对目标站点进行常规打点探测，过程中发现关联的小程序入口。初步浏览小程序仅看到名片展示功能，看似无高价值测试点，但在抓包分析接口数据包时，意外发现接口会返回其他用户的个人邮箱和手机号等敏感信息，同时注意到请求参数中存在一个疑似用户标识的 id 参数，由此初步判断存在越权漏洞风险。

随后尝试对该 id 参数进行常规 Base64 解密，结果解密内容无规律且无效，结合参数特征和经验进一步判断该 id 采用了 AES 加密方式。为获取加密密钥，直接对目标小程序进行反编译操作，拿到小程序源码后，通过搜索 “AES” 关键字定位到加密相关代码片段，从中提取出 AES 加密所需的 key、iv 向量、加密模式（CBC）及填充方式（Pkcs7）等核心信息，利用这些信息成功将加密的 id 参数解密为纯数字格式的原始用户 id。

接着编写自动化脚本，批量生成连续的原始用户 id，再用获取到的 AES 密钥将每个原始 id 加密为接口可识别的格式，批量发送接口请求进行遍历探测，最终成功获取到约 10 万条用户名片信息，包含手机号、邮箱等敏感数据。最后整理漏洞细节提交报告，厂商确认该漏洞为高危越权漏洞并完成修复。

0x03 Oracle 报错注入 WAF 绕过——垃圾数据 /**/ 注释干扰 + 手工脚本突破

处于未进入目标教育网站后台、准备开展漏洞探测的场景。思路是先拿到合法登录权限，才能访问内部接口并排查潜在漏洞。进行账号密码登录并抓取系统内接口请求数据包后，定位到 queryActivityList.do 可疑接口，重点核查接口参数是否直接拼接 SQL 语句，以此判断是否存在注入风险。

解析数据包后发现 sortField 参数会直接插入 SQL 语句的 ORDER BY 子句，可利用 Oracle 报错注入获取数据。但注入行为会被 WAF 拦截，且 ORDER BY 后参数长度受限。

核心绕过手法：在真实 SQL 注入语句前后，塞入超大量用 /**/ 注释符包裹的无用字符，这些被注释的垃圾数据会被数据库忽略，不影响注入语句执行，还能干扰 WAF 特征检测、避开 ORDER BY 后的长度限制，成功突破防护。

随后自动化注入工具 SQLMap 无法识别该漏洞，放弃工具依赖，自行编写 Python 脚本开展手动注入测试。逐步探测数据库结构后发现目标数据库存在一千多张数据表，且表名为汉字拼音首字母缩写，无法直观识别表内容。结合网站业务背景经验，人工推测表含义并逐个开展注入查询，排除空表后成功读取到有效业务数据。最后展示注入数据作为漏洞证明，向站点提出参数过滤、预编译查询等针对性安全优化建议。

0x04 水电费小程序未授权访问——若依（RuoYi）框架固定路径利用

进入目标大学宿舍电费微信小程序，随意绑定一个房间并开启代理抓取所有请求数据包。将核心请求包放入 Repeater 模块，手动修改接口路径为 /prod-api/system/user/list 并发送请求，成功访问用户列表接口，发现普通用户信息泄露。

继续修改接口路径为 /prod-api/system/config/list、/prod-api/system/user/1 并分别发送请求，探测到配置接口泄露管理员初始密码、用户详情接口泄露加密密码，确认接口存在严重未授权访问漏洞。

国内最常用的若依（RuoYi）框架，这套框架有固定死的后台接口地址，所有人都能用，不用瞎猜：

  • 用户列表：/system/user/list
  • 系统配置：/system/config/list
  • 部门列表：/system/dept/list

你只需要把抓到的任意接口前缀（如 /prod-api/）保留，后面直接套这些通用地址就能测试。所谓的 “经验”，只是知道「大学 / 企业后台最爱用若依框架」，知道这些固定地址而已，不是凭空乱改。

0x05 Spring Boot 敏感端点探测——/env 与 /heapdump 信息泄露利用

在 eduSRC 教育类站点的漏洞挖掘过程中，以目标系统登录页面为起点，首先开展弱口令、SQL 注入等常规 Web 漏洞测试。未发现明显漏洞后，转而分析前端 JS 文件以寻找潜在接口，但其暴露的接口均会重定向至登录页面，无法直接未授权访问。

随后通过页面图标特征识别出目标采用 Spring Boot 框架，便尝试使用 SpringBoot 专用工具批量探测该框架常见的 /actuator 系列敏感端点，然而工具未返回任何有效结果，推测工具特征被目标安全防护机制检测并拦截。

在此情况下放弃工具扫描，转为手工试探 Spring Boot 典型敏感路径：

  • 直接拼接访问 /env 端点，成功获取环境变量、内网地址等敏感信息
  • 结合经验判断存在 /env 端点通常会伴随 /heapdump 堆转储文件，进而访问并下载
  • 使用 JDumpSpider 工具对 heapdump 文件进行解密分析，最终提取出内网明文密码等高价值敏感数据

整个过程充分体现了 SRC 漏洞挖掘的核心思路：常规测试无果后需精准识别技术栈、灵活切换工具探测与手工测试模式；针对 Spring Boot 这类特定框架，在工具被防护时手工探测其特有敏感路径是突破关键；同时对泄露的 heapdump 等文件进行深度解析，能进一步挖掘出可直接利用的敏感信息，大幅提升漏洞挖掘的成功率与价值。

0x06 Nacos 未授权访问——配置泄露链式利用与第三方付费服务滥用

EDUSRC 漏洞挖掘以目标系统登录框为初始入口，先通过目录扫描工具对目标站点进行路径探测，成功发现暴露在外的 Nacos 服务接口。

随后借助 Nacos 相关 Nday 漏洞利用方式，直接访问 Nacos 用户列表接口获取用户数据，再使用专用工具对 Nacos 服务进行批量探测，确认目标存在 Nacos 未授权访问等高危漏洞。

为验证漏洞危害，先构造请求 Payload 执行未授权用户删除操作，确认可任意操作用户账号，还可进行未授权添加用户等类似操作。

进一步挖掘发现该 Nacos 服务支持未授权导出全部配置信息，导出的配置文件中大量泄露 MongoDB 数据库账户密码与地址、即时通信 IM 服务地址及账号密码、Redis 缓存单机及集群的密码、邮箱账号密码以及各类第三方服务密钥等核心敏感数据。

最后从泄露信息中定位到百度人脸识别服务的 APIKey 与 SecretKey，结合官方文档编写 Python 脚本调用该付费接口，验证漏洞可导致第三方付费资源被恶意滥用，以此完整证明该 Nacos 未授权漏洞从基础越权操作到核心数据泄露、再到高价值第三方资源滥用的全链条严重危害。

整体挖掘思路遵循从基础入口探测到敏感组件发现、从漏洞初步验证到深层数据提取、再到高危害场景落地利用的递进逻辑，逐步放大漏洞影响范围与危害等级。

0x07 微信小程序全链路漏洞挖掘——框架识别驱动专项突破与敏感数据批量遍历

在教育机构（EDU）场景下挖掘微信小程序漏洞，核心是瞄准学校微信小程序这一优质目标——这类小程序无需对接复杂的校园统一认证系统、功能模块高度集中（多围绕学生信息、教务管理、证书查询、校园服务等核心场景），且开发团队为快速上线往往存在大量安全疏漏，是漏洞挖掘的高价值阵地。

挖掘时首先通过微信搜索定位目标学校小程序，借助 Burp Suite 等工具抓取小程序与后端交互的所有请求流量，第一步先通过请求包路径、响应特征精准识别后端开发框架：

  • 若请求中出现 /prod-api、/xxx-api 前缀，可判定为若依（RuoYi）框架；
  • 若存在 /jeecg-boot/、/xxxx-boot/ 路径，则为 Jeecg 框架；
  • /api/sysxxxxxx 类接口多对应 Admin.net 框架；
  • SpringBoot 框架则可通过经典报错页面、/heapdump 敏感端点暴露或 /swagger 接口文档存在来识别。

识别框架后随即开展针对性漏洞挖掘：

若依（RuoYi）框架： 优先尝试 ruoyi/admin123、ry/admin123 等默认弱口令，若登录成功则进入后台利用定时任务功能构造恶意命令实现 RCE；同时在前台流量中检测 Shiro 框架特征（如 rememberMe Cookie），用 Burp 的 Shiro 插件爆破默认密钥——密钥成功则构造恶意 Cookie 触发反序列化 RCE，密钥失败则放弃该点；此外还要探测 Druid 监控接口是否存在弱口令，尝试登录获取数据库连接信息与 SQL 执行记录。

Jeecg 框架： 先通过微信一键登录获取 JWT 令牌，用令牌批量遍历各类敏感接口；同时尝试 Jeecg/123456 弱口令登录后台，测试积木报表功能是否存在 SQL 注入；还可查找普通用户提权至管理员权限的接口，无需管理员权限也能通过 JWT 越权获取敏感数据。

Admin.net 框架： 直接抓取小程序接口请求，批量梭哈 /api/sysxxxxxx 类接口，尝试越权获取学生证书、隐私信息等敏感数据。

SpringBoot 框架： 重点检测 /heapdump 端点是否开放，若开放则用 JDumpSpider 工具提取数据库密码、云服务密钥等敏感信息；同时遍历 /swagger 接口文档，逐个测试接口未授权访问漏洞，避免误删类高危接口。

完成框架专项挖掘后，再聚焦微信小程序特有漏洞场景深度挖掘：

  • 抓包分析微信一键登录流量，若发现 sessionkey、iv、加密数据泄露，可通过解密工具还原敏感信息，伪造目标用户手机号加密后发包，实现任意用户账号登录；
  • 检查接口请求参数，若 id、openid、unionid、phone 等参数可控，通过遍历 1-200 等数字区间，越权获取全校用户手机号、身份证、openid 等敏感信息，再利用部分小程序登录逻辑仅校验这三个参数的漏洞，直接接管全站用户账号；
  • 测试头像上传、文件提交等功能，检查是否未校验文件类型与后缀，尝试上传恶意脚本获取服务器权限；
  • 分析验证码验证逻辑：若验证码手机号明文写在数据包中，可修改为自己手机号接收验证码绕过校验；若为四位纯数字验证码则直接爆破 1000-9999 区间；
  • 爆破 13888888888 等常见测试手机号，往往能发现开发遗留的测试账号，进一步扩大漏洞范围。

最后遍历所有接口参数，修改 id、phone 等参数值，全面测试水平越权（访问同权限其他用户数据）与垂直越权（提升至管理员权限）。若能批量获取大量学生身份证、手机号等敏感数据，即可判定为高危漏洞。整套流程从框架识别、专项漏洞突破到小程序特有风险挖掘，形成完整的 EDU 场景微信小程序漏洞挖掘闭环。

0x08 教育系统组合拳——弱口令爆破 + SQL 注入 WAF 绕过 + 接口越权通杀

在教育机构SRC漏洞挖掘场景中，整个过程从前期信息搜集开始：先通过谷歌语法精准搜索目标学校域名下的默认密码、系统操作手册等公开信息，这类信息对应的老旧系统往往存在默认密码未修改的情况；同时利用谷歌语法、QQ群、各类社交平台搜集在读学生的学号信息，也可根据官网规则自编学号，而学生密码大多与身份证信息相关，为缩小数据范围可选取稀有姓名在电报平台筛选。凭借这些信息尝试登录系统，成功后便进入后续漏洞测试环节。

接着开展 XSS 漏洞测试，重点瞄准学校官网、二级学院网站中的投诉建议、留言反馈等存在输入框的功能点——只要有输入位置就插入测试字符，这类公开交互点大概率存在 XSS 漏洞，可作为基础漏洞提交获取分数。

随后聚焦 SQL 注入漏洞挖掘。第一个场景是学校教务一体化系统：先从贴吧等平台获取系统账号格式，再通过社群了解到默认密码规则为 账号+!qwer，据此构造弱口令字典进行爆破；成功登录后进入教室借用申请功能，前端直接输入测试字符无异常反应，于是抓包修改请求体数据，发现输入单引号后系统报错，再通过构造 1=1 返回数据、1=2 无数据的测试语句，确认存在 SQL 注入。针对 Oracle 数据库环境，使用 1'and 1=(case when ASCII(SUBSTRC(USER,1,1))=1 then 1 else 5 end) or '1'='2' 这类 payload 逐位爆破出当前数据库用户信息，同时发现该教务系统为多校通用平台，不同学校仅界面不同、接口逻辑完全一致，可实现漏洞通杀。

第二个场景是学校缴费平台的详情功能点：抓包后发现输入单引号系统报错、输入双引号则恢复正常，起初尝试参数污染、高并发请求、分段传输等多种 WAF 绕过手段均未成功；耗费大量时间后发现特殊逻辑——构造 1 in 1 请求会返回海量数据导致回显延迟，构造 1 in 5 请求则返回空数据。利用该特性先爆破出数据库用户名长度为 7 位，再通过 'OR+1+in+instr(user,'u')+and+'a'+in+'a 这类借助 instr 函数的 payload，逐位确认用户名的每一个字符，最终完成注入利用。

最后开展越权漏洞测试：在系统公告模块仅拥有普通查询权限、无删除新增权限，于是抓取搜索功能的请求包，分析接口命名规律，构造包含 delete、del 等关键词的接口路径字典，复用原有请求的完整数据发送测试，最终成功触发数据删除操作；同理也可通过构造对应接口测试越权新增等高危操作，完成整个教育机构系统的 SRC 漏洞挖掘流程。

0x09 新证书站抢时间挖洞——heapdump 泄露教师信息 → 默认密码横向突破 → 学生管理系统越权遍历

在 EDU SRC 平台上新证书站的漏洞挖掘场景中，由于证书上新后竞争极为激烈，漏洞容易被快速提交导致重复，因此挖掘核心思路是抢时间、重信息收集、从高风险通用漏洞入手逐步深入业务功能测试，具体过程如下：

当收到 EDU SRC 新证书站上线通知后，第一时间启动漏洞挖掘工作，先通过各类信息收集工具批量搜集目标院校相关域名与站点资产，摒弃耗时的师生信息登录测试，直接将整理好的域名分批导入目录扫描工具进行主动探测，同时手动浏览各站点初步了解业务形态。

扫描过程中重点关注到 Spring Boot 框架特有的 /actuator 接口可直接访问，确认存在 Spring Boot 未授权访问漏洞，进一步探测发现该接口下的 heapdump 文件可直接下载，随即使用专用工具分析该堆内存文件，提取出大量教师敏感信息，第一时间将该未授权访问漏洞提交至 EDU SRC 平台并获得中危评级。

后续深入测试时，先利用前期泄露的教师信息定位到学校统一身份认证系统，发现该系统明确标注默认密码为身份证号后六位，尝试使用已获取的教师身份信息成功登录三个教师账户，且登录后可无二次认证直接跳转到其他校内系统，测试后发现其中一个账户具备学生管理系统访问权限。

进入学生管理系统后切换至管理模式，对所有功能点进行抓包测试，在 “师生查询” 接口处发现存在敏感信息泄露问题，通过分析请求参数含义：

  • 修改 pageSize 参数扩大数据返回量
  • 将 inStudentStatusSign 参数设为 false 获取所有状态学生信息
  • 置空 nj 参数遍历所有年级数据

成功批量获取大量学生信息，后续还发现教师权限可直接导出全量学生数据。

为扩大挖掘范围，通过 Google 搜索语法查找该校其他存在默认口令的站点，利用已泄露的信息尝试登录校内邮箱等多个系统，虽多数系统强制登录后修改密码导致无有效信息泄露，但仍积累了更多可用信息。

随后再次回到学生管理系统，在登录页面发现 “忘记密码” 功能仅需账号、姓名和身份证号即可重置密码，利用此前获取的学生信息成功修改多个学生账号密码，登录学生账号后继续对个人信息等功能点抓包，发现学生个人信息查询接口路径直接使用学号作为参数，尝试替换学号后成功遍历获取所有学生的个人敏感信息。

经全面测试，学生账号下有 7 处、教师账号下有 2 处功能点存在类似遍历漏洞。

最后在权限测试环节，意识到系统权限控制存在缺陷，使用普通学生账号登录并抓包，将请求接口路径替换为教师权限对应的接口路径，成功绕过权限校验实现垂直越权，最终将所有发现的未授权访问、敏感信息泄露、任意文件下载、信息遍历、垂直越权等漏洞批量提交至 EDU SRC 平台。

整个挖掘过程充分体现了 “快速信息收集 → 优先挖掘高风险通用漏洞 → 利用泄露信息横向拓展 → 深入业务功能测试参数与权限 → 批量提交漏洞” 的核心思路，同时也凸显了在 EDU SRC 挖掘中，挖掘速度、全面的信息收集能力以及细致的功能点测试是成功获取漏洞的关键。

0x0A 一站式门户 IDOR 越权——userid 明文参数遍历 + 教师信息批量泄露

测试人员以普通学生身份登录某学院一站式服务大厅门户系统后，并未立刻使用自动化扫描工具，而是先以正常用户视角开展首轮信息摸排：仔细观察个人信息区域直接展示的学号、院系、班级等信息的格式规律，统计左侧功能菜单包含的子系统数量与涉及的业务类型，分析 URL 路径结构判断系统采用前后端分离的门户框架与 API 设计风格，同时留意页面加载的 JS、CSS 等静态资源，预判这类集成多业务模块的 edu 门户为了实现跨系统数据共享，往往会在权限控制上出现边界模糊的问题，且功能模块越多攻击面越大，个人信息区域大概率存在信息过度暴露风险。

完成初步观察后，测试人员开启 Burp Suite 代理进行流量监听，以正常操作的方式点击系统内各个功能模块，在 HTTP 历史记录中逐一梳理后台 API 的调用规律，重点关注带有 getUserInfo 等明显功能语义的接口路径，以及携带 userid、studentid 这类用户标识参数的请求，同时核对响应 JSON 数据与前端页面展示内容的差异。

基于 edu 系统开发常为了便捷直接返回完整数据对象的共性特征，判断查询类接口极易返回前端不需要的敏感字段，且带 ID 参数的接口大概率存在权限校验缺失的问题。

随后测试人员从捕获的接口中筛选出功能明确的用户信息查询接口进行深度分析，发现该接口采用 GET 请求方式，userid 参数以明文学号形式直接拼接在 URL 中，既无加密也无签名校验，进一步查看响应数据后发现接口返回了 SFZJH 字段且直接展示完整身份证号码，结合参数完全可控的特征，初步判定该接口可能未校验当前登录用户与查询目标的权限对应关系，存在水平越权漏洞。

为验证猜想，测试人员按照已掌握的学号格式规律构造其他用户的 userid 参数，修改原请求参数后重放数据包，发现系统返回 200 成功状态码，且响应体中返回了目标用户的完整个人信息与身份证号，证实系统仅根据传入的 userid 参数直接查询数据库，完全没有校验当前登录用户是否有权查看该用户信息，典型的 IDOR 越权漏洞确认存在。

在确认单点漏洞后，测试人员继续拓展信息收集范围，遍历系统其余业务模块，在在线课程与直播选课页面中意外发现，系统未经任何脱敏处理直接公开了全校 6000 余名教师的姓名、完整工号、所属院系等信息，且教师工号具备统一可预测的格式规律，这一信息泄露点恰好为已发现的越权接口提供了完整的批量目标 ID 列表，将单用户越权的危害直接放大为批量教师敏感数据泄露。

整个挖掘过程始终围绕 edu 系统的开发共性展开，遵循 “页面信息摸排 → 流量接口分析 → 单点漏洞验证 → 信息收集扩危害” 的思路，没有依赖高深的攻击技术，而是通过细致的信息收集将越权漏洞与信息泄露点组合利用，最终实现从普通学生账号到数千名教师身份证信息泄露的完整攻击链路，也充分体现出 edu 系统中对象级权限控制缺失、敏感字段全量返回、业务页面信息过度暴露三类问题叠加后的严重安全风险。

测试人员以普通学生账号登录某大学学生就业综合管理服务平台后，首先进入个人中心页面开展初步的功能与数据摸排，仔细查看页面展示的 2025 届毕业生分院系统计数据，包括各院系总人数、男女比例、往届对接人数等多维度汇总信息，同时梳理左侧功能菜单涵盖的用人单位信息查询、招聘会报名预约、简历维护、通知管理等业务模块。

结合就业系统统计查询需要灵活筛选条件、常通过动态拼接 SQL 语句实现的开发特点，预判数据统计类功能是 SQL 注入的高发区域，同时留意简历更新、信息查询、报表导出等涉及数据库交互的模块也可能存在输入验证缺失的问题。

在逐步测试系统各功能模块的过程中，当操作到就业数据统计相关功能时，页面出现异常跳转并返回错误提示，测试人员敏锐判断该异常大概率由后台 SQL 语句执行失败引发，随即开启 Burp Suite 代理对该操作对应的 HTTP 请求进行抓包分析，确认这是一个携带多参数的 POST 请求，所有查询参数均放置在请求体中传输，随后结合参数的命名语义与功能用途逐一排查用户可控输入点，筛选出直接参与后台数据查询条件构造的参数，判断该参数未经过严格的输入过滤与预编译处理，存在被直接拼接进 SQL 语句的可能。

在定位到可疑注入点后，测试人员使用 sqlmap 工具对该 POST 参数进行自动化注入检测与深度利用，最终成功获取到目标 Oracle 数据库的交互式 SQL shell，不仅可枚举到数据库内全部用户账号，包含系统内置用户与业务系统用户，还能直接执行自定义 SQL 查询语句，同时验证出当前数据库账号权限配置过高，为后续进一步获取全量业务敏感数据、深入内网渗透提供了基础条件。

整个挖掘过程遵循 “功能场景预判 → 异常现象捕获 → 流量分析定位 → 工具验证利用” 的思路，充分利用了业务系统统计查询功能的开发特性与错误反馈的信息泄露线索，逐步完成从漏洞发现到深度利用的完整链路。

0x0D EDU 内网攻防——H3C telnet 弱口令 → NAT 端口映射打点 → Redis 未授权反弹 → 密码复用横向拿下交换机

在本次 EDU 场景的漏洞挖掘过程中，攻击者拿到目标资产表后首先启动全量信息收集阶段，先用 fscan 对分配的目标做快速资产识别，初步扫过后发现可用外网资产不多，仅记录到一处内网地址泄露的信息，同时确认目标内网存在大量视频监控类设备。

在外网直接打点收益有限的情况下，攻击者转向弱口令突破路径，使用弱口令检测工具对暴露的服务做批量爆破，成功爆破出一台 H3C 网络设备的 telnet 管理员口令，登陆设备后发现其自带 SSL VPN 功能与授权，最初的思路是通过配置 SSL VPN 直接打通内网访问隧道，但该设备型号过于老旧，无论是命令行参照官方手册配置，还是通过 web 界面调试，都始终无法成功建立 VPN 连接。

在这个单点耗费近一小时无果后，攻击者及时调整攻击思路，转而利用已拿到的路由器管理员权限做进一步内网探测，先通过查看设备路由表梳理出内网 2、3、5、6 共四个网段，再借助设备自带的 telnet、ping 能力探测网段首尾 IP 的端口开放情况，以此判断出 2 网段为服务器区、5 网段包含服务器与无线 AP 设备，3 网段属性暂不明确。

随后攻击者登陆了 5 网段的一台无线 AP，发现设备仅支持 ifconfig、ps 等少量命令，其余命令均被屏蔽，web 端功能也十分有限，未找到可利用漏洞，查阅厂商手册得知同系列无线网关支持 VPN 功能后尝试寻找网关设备但未能定位，便彻底放弃 VPN 打通的路径，改用 “端口映射 + 脆弱端口探测” 的路径。

依托路由器的 NAT 端口映射能力，针对已确认的服务器网段 2 段，逐个探测 22、1433、3389、6379、7001 等常见脆弱端口，每发现开放端口就通过路由器将其映射到公网做后续利用。最开始探测 22 端口做 SSH 弱口令爆破全程无果，继续扩大探测范围后发现一台主机开放 6379 Redis 端口，在路由器 telnet 环境下验证未授权访问表现异常，将端口映射到公网后确认该 Redis 存在未授权访问漏洞，随即使用 fscan 通过 Redis 写入计划任务的方式反弹 shell 到公网 VPS，成功拿到第一台内网主机权限，撕开了内网突破口。

进入内网横向阶段后，攻击者沿用密码复用的思路，用已收集到的密码通过 fscan 批量撞库内网 B 段的 3389 远程桌面端口，成功拿下两台主机权限，同时在内网批量探测向日葵 RCE 漏洞，筛选出可正常执行命令的主机后用 PowerShell 将其上线到 CS。

登陆服务器后按常规路径翻查本地文件、导出浏览器保存的密码、检索管理员登陆日志定位管理端 IP，利用找到的数据库账号成功连入数据库，还从浏览器凭据中获取到上网行为管理系统的账号密码并登陆后台，除此之外，将已获取的密码批量尝试内网交换机设备的登陆权限，最终拿下 8 台交换机的控制权。

后续攻击者又用 kscan 做更全面的内网端口扫描，发现大量移动终端开放 ADB 端口，批量连接后覆盖了物联网类资产的得分上限。

全程中攻击者多次调整攻击思路，从死磕 SSL VPN 配置转向端口映射打点，避免在单一攻击路径上过度消耗时间，同时也总结出 EDU 场景下信息收集要尽可能宽泛全面、内网横向要善用密码复用、遇到瓶颈及时切换突破口的挖掘经验，也意识到团队协作中信息同步、多思路碰撞能有效避免钻牛角尖，提升漏洞挖掘的效率。

0x0F 后勤管理系统漏洞链——弱口令直取超管 → Cookie 替换绕过前端权限拦截 → 删除账号不失效的隐形后门

本次针对高校资产的漏洞挖掘始于常态化的脆弱资产测绘，攻击者在周末通过 ARL 灯塔工具梳理高校域名资产，摒弃信息泄露类的常规挖掘方向，转而在站点标题维度检索 “系统、注册、登录” 等关键词快速定位脆弱目标，最终锁定了一个校园后勤类 xx 管理系统的登录入口。

面对登录框，攻击者结合 EDU 场景的漏洞价值特性，先排除了 URL 重定向、反射型 XSS 这类低价值 “水洞”，优先测试弱口令、未授权访问、信息泄露、返回包绕过等高效突破思路，在观察到页面默认回显账号为 qyxx、密码栏呈现四位掩码的特征后，基于高校后勤管理系统常存在账号与密码一致的弱口令配置惯例，直接推测密码与账号同名，使用 qyxx/qyxx 的凭据尝试登录，成功进入系统后台。

且该账号是系统唯一的超级管理员账户，拥有校园内吹风机、洗衣机等共享设备的开关控制、消费金额调整等全部管理权限，后台还存储着数千条居民身份证等敏感个人信息，仅这一处弱口令漏洞就达到 6rank 的评级。

拿到超管权限后，攻击者继续深度挖掘后台的权限控制缺陷，针对垂直越权开展测试：

  1. 以超管身份进入操作员添加功能页，创建仅授予运营模式管理权限的低权限 admin 账户，同时抓取创建用户时的完整 HTTP 请求包存入 Burp 的 Repeater 模块
  2. 切换到 admin 账号登录，验证前端界面已屏蔽新增用户的功能入口、低权限用户无创建用户的前端权限
  3. 按照垂直越权的标准测试逻辑，将请求包中原超级管理员 qyxx 的 Cookie 替换为低权限 admin 的 Cookie 后重放请求，成功创建出 admin1 用户

证实系统后端未对用户权限做校验，仅靠前端做权限拦截，存在垂直越权漏洞，且该漏洞可延伸至删除用户、金额管理等所有超管功能点，甚至可以先抓取超管创建最高权限账户的请求包，再用低权限账号的 Cookie 重放请求，直接越权生成超级管理员账户实现权限完全逃逸。

完成越权漏洞验证后，攻击者进一步测试系统的会话管理逻辑：

  1. 以超管身份创建 admin2 测试账号，在另一浏览器中使用 admin2 完成登录保持会话
  2. 回到超管后台将 admin2 账号从用户列表中删除
  3. 切换到 admin2 的登录页面刷新，发现账号不仅没有被强制下线，还能正常查看所有敏感信息
  4. 甚至可以继续借助垂直越权的逻辑创建新的 admin3 用户，且新创建的 admin3 账号也能正常登录系统

由此验证系统存在严重的会话逻辑缺陷：用户被管理员删除后，只要客户端不主动退出登录，已签发的会话 Cookie 就始终保持有效、永不过期。攻击者可以利用这一缺陷创建高权限账号后主动让管理员删除，留存一个管理员在用户列表中完全感知不到的隐形后门账户，长期维持对系统的控制。

整套挖掘流程从资产定位到弱口令突破，再到后台权限与逻辑漏洞的纵深挖掘，最终收获一个高危加两个中危的漏洞评级。
