前言：颠覆对电子邮件地址的认知

常规认知中，邮箱格式为本地部分@域名，系统仅做简单格式校验，但真实情况为：

  1. 邮箱解析无统一标准，完全依赖解析库、邮件服务器的自定义规则；
  2. 解析库为轻量级字符串处理，仅做基础匹配运算，不会消耗大量硬件资源；
  3. 邮箱RFC协议为兼容老旧技术，保留大量冗余、宽松的解析规则；
  4. 解析规则不统一+协议过度宽松+第三方库违规实现，催生了域名验证绕过、内网入侵、企业服务器控制等高危漏洞。

本文将跟随研究者思考路径，从邮件路由异常切入，逐层揭开邮箱解析漏洞的完整体系。

第一章 基础铺垫：邮箱解析核心知识

1.1 理论标准与现实解析差异

邮箱官方标准遵循RFC 5321/5322，格式为本地部分@域名，理论支持特殊字符、注释、转义、上古协议格式，但现实中无系统完全遵守该标准。

  • 前端：仅通过简单正则校验邮箱格式；
  • 后端：依赖编程语言解析库（如Ruby Mail Gem）处理邮箱；
  • 邮件服务器（MTA）：Sendmail、Postfix负责邮件路由；
    三方解析规则不一致，是所有漏洞的根源。

1.2 邮箱解析库的核心原理

所有系统的邮箱校验均依赖解析库，无手写解析的可行性，核心特性为：

  1. 采用轻量级分层校验，仅执行字符串查找、分割、正则匹配，无复杂运算；
  2. 规则模块化设计，按需执行校验逻辑，非全量规则遍历；
  3. 解析规则差异源于RFC标准、厂商自定义、历史协议兼容，也是漏洞产生的核心原因。

1.3 核心邮件服务器（MTA）

  • Sendmail 8.15.2：老牌邮件服务器，兼容UUCP上古协议，解析规则臃肿且宽松；
  • Postfix 3.6.4：现代主流邮件服务器，安全性更优，但仍保留源路由、百分号Hack等老旧兼容规则；
    二者为兼容旧协议，均保留了存在安全风险的解析逻辑。

第二章 研究起点：邮件路由劫持——Sendmail与Postfix的解析异常

研究者的研究始于一个意外现象：构造畸形邮箱地址后，邮件未发往目标example.com，反而被路由至攻击者控制的oastify.com。

2.1 Sendmail 8.15.2：UUCP协议引发的路由劫持

研究者思考过程

测试Sendmail时发现，其为兼容UUCP（Unix间拷贝协议），支持域名!用户的古老地址格式，该格式为现代邮箱标准所不支持。

漏洞Payload

解析过程

  1. 反斜杠\转义原生@，导致Sendmail无法识别标准邮箱格式；
  2. Sendmail将!识别为UUCP协议的地址分隔符，遵循目标域名!账户名的解析规则；
  3. 服务器判定目标域名为攻击者控制的oastify.com，直接忽略原目标example.com；
  4. 邮件被错误路由至攻击者服务器，实现邮件路由劫持。

2.2 Postfix 3.6.4：路由劫持的漏洞变体

研究者思考过程

初始判定Postfix无UUCP协议兼容，安全性更高，遂编写SMTP模糊测试器批量测试特殊字符，意外发现同源路由劫持漏洞，仅触发方式不同。

核心漏洞原理：百分号Hack + 括号注释规则

  1. 百分号Hack（源路由协议兼容）：Postfix会将地址中的%自动转换为@，为邮件中继功能的遗留规则；
  2. 括号注释规则：遵循RFC协议，将()内的内容视为注释直接忽略，且未闭合的单括号可触发后续内容全量丢弃。

漏洞Payload

解析过程

  1. 单括号(触发注释规则，Postfix将后续example.com全部视为注释并丢弃；
  2. 剩余内容collab%psres.net被识别为源路由地址；
  3. 按百分号Hack规则，%转换为@，最终解析为collab@psres.net；
  4. 邮件被路由至攻击者控制的psres.net，路由劫持成功。

关键发现

单括号无需闭合即可截断域名，是Postfix解析器的核心校验缺陷。

第三章 突破前端拦截：Unicode溢出——伪造被禁止的特殊字符

3.1 研究者遇到的核心问题

实战中发现，大量Web应用通过正则直接禁止多个@等特殊字符，无法直接构造畸形邮箱地址，需找到前端放行、后端可还原为被禁字符的伪造方法。

3.2 Unicode溢出核心原理（基于PHP chr()算法）

ASCII字符的编码范围仅为0~255，PHP的chr()函数对输入的数值执行模256运算，将超大数值强制压缩至ASCII编码范围，这一特性即为Unicode溢出。

核心计算公式

目标ASCII字符 = 超大Unicode码点 % 256

典型案例：伪造@符号

  • @符号的ASCII编码：0x40（十进制64）；
  • 构造超大Unicode码点：0x100（十进制256）+ 0x40 = 0x140（十进制320）；
  • 运算过程：320 % 256 = 64，后端解析后直接生成@符号。

3.3 模糊测试与通用规律

研究者通过Turbo Intruder工具模糊测试后，总结出Unicode溢出的通用规律：

  1. 只要Unicode码点满足码点%256=目标ASCII值，即可伪造任意被前端拦截的字符（@、!、%、>等）；
  2. 从0x100到Unicode最大码点0x10ffff，海量字符均可实现该溢出效果；
  3. 前端将超大Unicode字符识别为正常非ASCII字符并放行，后端通过模运算还原为恶意ASCII字符。

3.4 研究意义

虽未在首个测试目标上直接实现邮箱地址拆分，但证明了被拦截特殊字符的伪造可行性，为后续突破前端防护扫清了核心障碍。

第四章 核心绕过手段：Encoded-word编码——多层编码套娃绕过校验

4.1 研究者的关键发现：RFC 2047 Encoded-word编码

翻阅RFC协议时发现，RFC 2047定义的Encoded-word编码，其设计初衷为给邮件标题、发件人昵称编码非英文字符，协议明确规定不可用于邮箱地址本身。

4.2 Encoded-word标准编码格式

  • =?：编码开始标记；?=：编码结束标记；
  • 支持字符集：UTF-8、ISO-8859-1、已废弃的UTF-7；
  • 支持编码方式：q=Q编码（十六进制编码，=40对应@）；b=Base64编码。

基础解码示例

解码后结果：ABCUSER@psres.net

4.3 致命漏洞：Ruby Mail Gem的违规解码

研究者思考过程

通过Encoded-word格式测试大量网站后发现，所有存在解析漏洞的网站均基于Ruby开发，且均依赖下载量超5.08亿次的Mail Gem解析库。

该库的核心违规行为

  1. 违反RFC 2047协议，将Encoded-word解码逻辑非法套用在邮箱地址解析中；
  2. 仍支持已废弃的UTF-7编码，且允许UTF-7+Q编码+Base64的多层混合编码；
  3. 前端无法识别编码后的恶意邮箱格式，后端解析库会自动完成多层解码，还原为恶意地址。

验证标准：若目标服务器解码后，向攻击者服务器发送SMTP交互请求，即证明存在该解析漏洞。

第五章 实战攻坚：三大顶级平台漏洞解析——试错与Payload重构全流程

研究者将Unicode溢出、Encoded-word编码、SMTP协议特性结合，针对Ruby开发的主流平台展开实战，以下为完整的尝试→失败→分析→Payload重构→成功全过程。

5.1 GitHub：绕过Cloudflare Zero Trust，访问企业内网

目标背景

GitHub基于Ruby开发，Cloudflare Zero Trust将其作为身份提供商（IdP），仅允许白名单域名的邮箱账户访问企业内部网络。

研究者试错与重构过程

  1. 首次尝试：在引号内嵌入原生@、通过Q编码构造双引号（=22）拼接畸形地址，均失败；
  2. 失败原因：地址虽通过前端验证，但邮件被服务器前置过滤，无任何SMTP交互反馈；
  3. 思路突破：挖掘SMTP协议核心指令RCPT TO的特性——>可直接结束收件人指令，忽略后续所有内容，补充空字符=00可避免后端解析异常，彻底截断原域名解析逻辑；
  4. Payload重构：放弃引号嵌套思路，直接用Q编码生成核心控制字符，组合为可截断SMTP指令的恶意地址。

关键控制字符编码

  • =40 → 解码为@，构造恶意收件人；
  • =3e → 解码为>，结束SMTP的RCPT TO指令；
  • =00 → 解码为空字符，让服务器忽略后续内容，避免解析异常。

最终成功Payload

解析过程与漏洞危害

  1. 后端解析后生成@>，直接截断SMTP指令，邮件服务器仅识别攻击者控制的域名；
  2. 可在GitHub上验证任意域名的邮箱账户（microsoft.com、github.com等），无需实际拥有该域名；
  3. 利用GitHub的IdP身份，直接绕过Cloudflare Zero Trust防护，访问受保护的企业内部网络。

5.2 Zendesk：绕过支持中心的域名白名单限制

目标背景

Zendesk支持中心可配置域名白名单，仅允许指定企业域名注册账户，其核心架构基于Ruby开发。

研究者试错与重构过程

  1. 首次尝试：直接复用GitHub的Payload，解析后无任何截断效果，漏洞利用失败；
  2. 失败原因：Zendesk的解析规则、邮件服务器与GitHub存在差异，核心控制字符被系统直接过滤；
  3. 思路突破：通过双攻击者域名测试发现Zendesk的解析特性——会自动删除地址中的异常字符，可利用该特性完成引号闭合，间接拆分邮箱地址；
  4. Payload重构：构造含异常字符的编码内容，利用系统的自动删除特性完成语法闭合，再复用截断逻辑。

最终成功Payload

解析过程与漏洞危害

  1. =3c22解码为<"，系统自动识别<为异常字符并删除，仅剩双引号完成地址语法闭合；
  2. =3e=00复用SMTP指令截断逻辑，成功拆分邮箱地址；
  3. 绕过支持中心的域名白名单限制，实现未授权注册与访问。

5.3 GitLab：未授权访问企业版服务器

目标背景

GitLab基于Ruby开发，企业版支持严格的域名注册限制，分为私有化版（搭配Postfix邮件服务器）和Web在线版，二者解析规则存在差异。

研究者试错与重构过程

  1. 首次尝试：直接复用GitHub、Zendesk的Payload，均被GitLab的解析规则拦截，利用失败；
  2. 失败原因：GitLab对编码字符、特殊符号的校验更严格，直接的指令截断、语法闭合逻辑均被阻断；
  3. 思路突破：回溯Encoded-word协议特性——下划线_可被解析为空格，且GitLab解析库对该规则完全兼容，比编码空格=20更易绕过过滤；
  4. Payload重构：在编码内容中加入下划线_，适配GitLab的解析特性，再组合核心控制字符；同时针对私有化版和Web版的解析差异，微调核心字符。

最终成功Payload

版本适配与漏洞危害

  1. 私有化版：直接使用上述Payload，下划线_的解析特性可完美绕过过滤；
  2. Web在线版：将下划线_替换为编码空格=20，即可实现漏洞利用；
  3. 绕过域名注册限制，未授权登录GitLab企业版服务器，获取私有代码、内部项目权限，甚至可利用GitLab的IdP身份，进一步入侵关联系统。

第六章 延伸攻击：从邮箱解析漏洞到服务器远程代码执行（Joomla实战）

研究者将研究范围延伸至Punycode国际化域名漏洞，结合邮箱解析漏洞，实现了从应用层权限绕过到**系统层远程代码执行（RCE）**的完整攻击链。

6.1 Punycode漏洞核心原理

国际化域名会被转换为xn--*格式的Punycode编码，部分系统仅校验前端展示的Unicode域名，未校验后端实际解析的Punycode域名；结合Unicode溢出特性，可篡改域名解析结果，注入恶意字符。

6.2 Joomla完整攻击流程

  1. 注入畸形字符：在Joomla的邮箱字段中，注入畸形Unicode/Punycode字符，利用解析漏洞绕过域名校验，将恶意代码嵌入系统存储的邮箱地址中；
  2. 触发注入漏洞：系统在后台、用户列表等页面直接输出邮箱地址时，畸形字符破坏页面HTML结构，触发CSS/HTML注入；
  3. 权限提升：通过CSS/HTML注入窃取管理员Cookie、会话信息，获取后台最高权限；
  4. 实现RCE：结合Joomla的模板编辑、插件上传漏洞，上传恶意代码至服务器，实现远程代码执行，完全控制目标服务器。

6.3 完整攻击链总结

邮箱解析漏洞 → 域名校验绕过 → CSS/HTML注入 → 管理员权限提升 → 服务器远程代码执行，漏洞危害从应用层直接升级至系统层。

第七章 漏洞根源复盘与防护方案

7.1 所有漏洞的核心根源

  1. 协议过度宽松：RFC邮箱协议为兼容UUCP、源路由等老旧协议，保留了大量冗余的解析规则，为漏洞提供了基础；
  2. 解析规则不统一：前端正则、后端解析库、邮件服务器的三方解析规则完全不一致，存在大量可被利用的规则差异；
  3. 第三方库违规实现：Ruby Mail Gem是大厂中招的核心原因，其违反RFC协议，将Encoded-word解码逻辑套用在邮箱地址解析中；
  4. 校验流于形式：绝大多数系统仅通过简单正则校验邮箱格式，忽略了编码、溢出、特殊字符的安全风险。

7.2 针对性防护方案

  1. 严格限制邮箱格式：仅允许标准ASCII字符，禁用Unicode字符、Encoded-word编码格式；
  2. 修复解析库漏洞：禁止解析库对邮箱地址执行Encoded-word、UTF-7等非标准解码操作；
  3. 统一解析规则：让前端、后端、邮件服务器使用完全一致的邮箱解析规则，消除规则差异；
  4. 关闭老旧兼容规则：在Sendmail、Postfix中，禁用UUCP、百分号Hack等非必要的老旧协议兼容规则；
  5. 过滤核心危险字符：在邮箱校验中，拦截@、>、(、)等可触发SMTP指令截断的特殊字符。

第八章 研究者的完整思考链路

  1. 偶然发现Sendmail的邮件路由异常，验证出UUCP协议的路由劫持漏洞；
  2. 编写SMTP模糊测试器，在Postfix中发现路由劫持的漏洞变体，打破Postfix“绝对安全”的认知；
  3. 针对前端的特殊字符拦截，研究PHP chr()算法的Unicode溢出特性，实现被禁字符的伪造；
  4. 挖掘RFC 2047 Encoded-word编码规则，发现Ruby Mail Gem的违规解码漏洞，找到核心绕过手段；
  5. 结合SMTP协议特性，总结出指令截断技巧，构造出通用的恶意Payload；
  6. 针对Ruby开发的GitHub、Zendesk、GitLab展开实战，根据各平台的解析规则差异，完成Payload的个性化重构；
  7. 延伸研究Punycode国际化域名漏洞，结合邮箱解析漏洞，实现从权限绕开到服务器RCE的完整攻击链；
  8. 总结所有漏洞的核心根源、通用利用向量，提出针对性的防护方案，完成整个研究。

结语

本项研究的核心价值，在于揭示了一个被普遍忽略的事实：看似标准化的邮箱地址，背后是混乱的解析规则和大量的安全漏洞。

对于安全初学者而言，该研究也印证了安全研究的核心逻辑：漏洞并非只存在于复杂的功能模块中，更多藏在被所有人忽略的基础协议、解析细节里。而邮箱解析的规则差异、第三方库的违规实现，也警示开发者：基础功能的校验并非“简单格式匹配”，统一规则、严格校验，才是防范安全漏洞的核心。
