浏览器同源策略与CORS解析

post @ 2026-04-12

前言：为什么需要同源策略？

互联网诞生之初，浏览器允许任意网页随意访问其他网站的数据。这带来了巨大的安全隐患：恶意网站可以悄悄读取你在银行网站上的个人信息，甚至冒充你进行操作。为了解决这个问题，浏览器引入了 同源策略，作为一道基本的安全防线。

但同源策略过于严格，现代 Web 应用又需要合法的跨域通信（比如前端调用不同域名的 API）。于是，W3C 制定了 CORS，在保证安全的前提下，为跨域访问打开一扇可控的门。

本笔记将按照 “问题 → 解决方案 → 细节机制 → 安全风险” 的逻辑，为你彻底讲清同源策略与 CORS。

1. 同源策略：浏览器安全的基石

问题由来

如果没有同源策略，当你登录银行网站 https://bank.com 后，再打开一个恶意网站 https://evil.com，恶意网站可以这样攻击：

由于浏览器会携带银行网站的 Cookie，请求会成功返回你的账户数据，然后被恶意网站窃取。

JWT

post @ 2026-04-10

0x.00.JWT基础知识

  1. 🧩 JWT 的核心结构

一个 JWT 由三部分组成，用 . 分隔：

部分 名称 说明
第一段 Header（头部） 包含令牌类型（typ: "JWT"）和签名算法（如 HS256、RS256）。Base64Url 编码。
第二段 Payload（载荷） 包含声明（claims），如主题声明（sub）、过期时间（exp）、签发时间（iat）等。Base64Url 编码。
第三段 Signature（签名） 对前两部分进行签名，防止篡改。

  1. 注册声明

    • 官方声明
      • iss (issuer)：签发方
      • iat (issued at)：签发时间
      • sub (subject)：主题（一般放用户唯一标识）
      • aud (audience)：受众
      • exp (expiration time)：过期时间
      • nbf (not before)：在此时间之前不可用
      • jti (JWT ID)：唯一编号，防重放
    • 公共声明（Public Claims）:由 IANA 统一登记维护，行业通用、无冲突风险，不属于官方注册声明但广泛使用
      • auth_time：用户实际完成身份认证的时间
      • nonce：随机值，多用于 OIDC 场景，防止重放攻击
      • azp：授权参与方，通常存储客户端 ID
      • scope：权限范围字段，定义令牌可访问的资源范围
      • client_id：调用接口的客户端唯一标识
      • sid：会话 ID，关联用户当前登录会话
      • at_hash：访问令牌哈希值，OIDC 中用于校验令牌一致性
    • 私有声明（Private Claims）:业务方自定义字段，无统一标准，JWT 渗透测试中最核心的篡改目标
      • role/roles：用户角色，常见值 user/admin/root
      • username：登录用户名
      • userId/uid/id：用户唯一标识 ID
      • admin：布尔型标识，true 代表管理员权限
      • permissions：权限数组，存储具体操作权限
      • email：用户邮箱
      • group：用户所属分组
      • is_admin：管理员权限标识

  2. ✅ JWT 的典型使用流程

    • 登录：客户端发送用户名/密码 → 服务器验证成功 → 生成 JWT 并返回。
    • 存储：客户端通常将 JWT 保存在 localStorage、sessionStorage 或 HttpOnly Cookie 中。
    • 请求：客户端在后续请求的 Authorization 头或是Cookie中携带 JWT
    • 验证：服务器收到 JWT 后，验证签名、有效期、issuer 等，从载荷中提取用户身份。

  3. 📦 JWT vs Session（传统会话）

特性 JWT Session
服务端存储 无状态（不存储） 有状态（存储会话数据）
扩展性 易于水平扩展 需要共享存储（如 Redis）
吊销能力 困难（需黑名单） 简单（删除会话记录）
载荷大小 较大（包含用户信息） 很小（仅 session ID）
  1. 流程图


实现身份认证依靠的是检查 令牌的完整性（未被篡改）和真实性（由服务器签发），就会设置JWT的比对机制

对称签名算法(HS256，即 HMAC+SHA256)：同一个密钥既用于生成签名，也用于验证签名。这个密钥必须绝对保密，不能让客户端或任何第三方知道。若服务器可以使用所存处的密钥成功对应客户端发来的jwt进行解密，则通过认证。
