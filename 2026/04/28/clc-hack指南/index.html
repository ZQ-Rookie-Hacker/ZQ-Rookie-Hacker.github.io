4.3.1 补充：为什么不能用 /skill-name 调用？

如果你尝试输入 /sqli-sql-injection 或 /hack，会得到 “Unknown skill” 错误。这不是 bug，是架构设计：

原生 Skill 系统 Hack Skill 系统
目录 ~/.claude/skills/ ~/.cc-switch/skills/（CC Switch 管理）
调用方式 /skill-name 斜杠命令 → Skill tool 自然语言输入 → Hook 自动推荐 → Read tool
触发机制 Claude Code 内置调度 UserPromptSubmit Hook 拦截 + 关键词匹配
管理方式 手动放置 / /plugin install CC Switch GUI 安装/更新/同步 + Hook 路由

关键点：

  1. 技能由 CC Switch 统一管理（~/.cc-switch/skills/），GUI 界面安装/更新/同步，无需手动 git clone
  2. Hook 输出中的 "不要调用 Skill tool（已从 skills 目录移除）" 是固定提示文案（skill-router-hook.py:328），每次匹配到安全任务都会显示，不是诊断信息
  3. 这样设计的优势：CC Switch 管静态（安装/更新/同步） + Hook 管动态（关键词路由/使用追踪/双语匹配），比原生 skill 系统的静态注册更灵活

正确使用方式：直接输入安全相关问题（如 “帮我测这个 SQL 注入”），Hook 自动推荐最相关的 SKILL.md 路径，Claude 通过 Read tool 按需读取即可。

4.6 鲁棒性加固要点（v4.5 / v3.0）

以下是在实际使用中暴露的故障点及根因修复，供参考：

Hook 1: security-context-hook.py

故障点 根因 修复
用户输入含 { } 时崩溃 str.format(prompt=...) 将花括号视为格式占位符 → KeyError 改用 %s 字符串替换
网络抖动一次就降级 零重试机制 2 次重试 + 递增退避 (1s, 2s)
不同 API 厂商响应格式不同 假设 data["content"][0]["text"] 一定存在 _parse_llm_response() 防御式解析：检查 content 类型、过滤 type=="text"、检查 error 字段、检查 stop_reason
正则每次调用重新编译 @lru_cache 但函数内无参数 改为模块级预编译 _COMPILED_SECURITY / _COMPILED_SCENES
超大输入跑满 CPU 无输入大小限制 _MAX_KEYWORD_SCAN = 8192 只扫描前 8KB
timeout 环境变量非数字 int(os.getenv(...)) → ValueError try/except + 限幅 3-30s
异常时 print(raw) 含 surrogate _scrub() 未覆盖所有路径 统一 _safe_print() 包装所有输出
sys.stdout.reconfigure 不存在 旧版 Python (<3.7) 无此方法 try/except (AttributeError, OSError): pass
Status line 写文件静默失败 _write_hook_status 使用 tempfile 但缺少 import v4.4: 添加 import tempfile
Auto 模式 LLM 返回通用场景时 scene 丢失 LLM 分类返回 scene=None 后短路 keyword 补充 v4.4: auto 分支 LLM 未确定 scene 时用 keyword 结果补充

Hook 2: skill-router-hook.py

故障点 根因 修复
stdin 含 surrogate 字符 缺少 surrogateescape sys.stdin.reconfigure(errors="surrogateescape")
文件写入竞争 多个 hook 实例并发写 .skill-usage.json _safe_save() atomic write (tempfile + os.replace)
索引重建失败静默 importlib.util 动态加载无防御 检查 spec is None / loader is None
_is_security_task 正则每次编译 re.search(kw, ...) 每次调用 预编译 _COMPILED_SECURITY_KW
超大 prompt 慢速匹配 prompt.lower() + 全量正则扫描 _MAX_KEYWORD_SCAN = 16384 + 分阶段截断
输出含非法字符崩溃 无 surrogate 清理 统一使用 _safe_print()
usage.json 损坏时崩溃 json.load() → 未检查返回值类型 防御：检查 isinstance(data, dict)
Status line 写文件与 Hook 1 竞争 两个 Hook 同时读写 .hook-status.json v2.2: 各自 read-merge-write + atomic replace，字段独立（scene/skills）不互相覆盖

现象 原因 解决
Hook 不触发 Python 不在 PATH 中 确认 python --version；settings.json 中使用完整路径 C:/Python311/python.exe
Hook 执行报错 脚本编码/路径问题 设置 CLAUDE_HOOK_DEBUG=1；检查路径是否用正斜杠
UserPromptSubmit hook error hook 配置缺少 matcher 字段 settings.json 中 UserPromptSubmit hooks 必须使用 matcher + hooks 格式：每个 hook 组需要 "matcher": "" 字段（空字符串 = 匹配所有输入），并将多个 command 放在同一组的 hooks 数组中

UnicodeEncodeError: surrogates not allowed 输入中包含 lone surrogate 字符（如 \udcaf），Python UTF-8 编码器拒绝输出 修复脚本：sys.stdout.reconfigure(encoding="utf-8", errors="surrogateescape") + 统一使用 _safe_print() 包装所有输出
只有 Hook 1 工作 Hook 2 依赖文件缺失 确认 keyword-aliases.json 和 build-skill-index.py 已安装；运行 python ~/.claude/hooks/build-skill-index.py
LLM 分类不工作 无 API Key 或网络不通 Hook 自动降级为关键词模式，无需额外操作
只想用关键词模式 默认是 auto 设置环境变量 CLAUDE_HOOK_MODE=keyword
Claude 仍拒绝 Rules 路径不对 确认文件名为 rules--security-research-context.md，位于 ~/.claude/rules/common/
输入 /sqli-sql-injection 提示 Unknown skill hack skills 不走原生 Skill 系统，~/.claude/skills/ 目录不存在 正常现象。直接输入安全关键词，Hook 会自动推荐对应 SKILL.md；详见 4.3 补充说明
每次都提示 “不要调用 Skill tool（已从 skills 目录移除）” skill-router-hook.py:328 固定文案 正常现象。这是 hook 脚本内置提示，告知用 Read tool 而非 Skill tool 读取推荐的 SKILL.md
状态栏不显示或显示乱码 Python 编码问题或 status-line.py 权限 确认 python C:/Users/你的用户名/.claude/hooks/status-line.py 可正常执行；检查 .hook-status.json 是否生成
