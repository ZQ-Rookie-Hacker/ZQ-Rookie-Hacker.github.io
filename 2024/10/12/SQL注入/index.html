关于information_schema库

MySQL 从 5.0.2 版本开始正式引入了 INFORMATION_SCHEMA 数据库。

关键细节

  1. 首次出现版本：5.0.2（2005年发布）
  2. 目的：
    提供符合 SQL 标准（ISO/IEC 9075）的元数据访问方式，替代部分 SHOW 命令（如 SHOW DATABASES、SHOW TABLES）和直接查询 mysql 系统表的方式。
  3. 兼容性：
    所有 MySQL 5.0 及更高版本（包括 5.1、5.5、5.6、5.7、8.0）均默认包含此库。

为什么引入？

  • 标准化：遵循 SQL 2003 标准，提升跨数据库兼容性。
  • 安全性：通过视图而非直接系统表暴露元数据，更安全。
  • 灵活性：支持通过 SQL 语句（SELECT）查询元数据，便于程序化处理。

版本演进

MySQL 版本 是否支持 说明
< 5.0.2 ❌ 使用 SHOW 命令或直接查询 mysql 库中的系统表
≥ 5.0.2 ✅ 默认启用，持续增强视图内容（如 8.0 新增索引统计视图）
≥ 8.0 ✅ 优化性能，部分元数据改用数据字典表存储

注意

  • 在 MySQL 8.0 中，部分底层元数据存储从 MyISAM 表改为 InnoDB 数据字典表，但 INFORMATION_SCHEMA 仍通过视图提供兼容接口。
  • 若您在使用 4.x 或更早版本（已过时），需升级到 5.0+ 才能使用该功能。

建议升级到受支持的版本（如 5.7 或 8.0）以获取完整功能和安全更新。

二、SQL注入类型总结

前言

1.在SQL注入中，各种注入类型的边界比较模糊，就比如自然数1属于实数集R，它也属于整数集Z。实际上你如何理解这种类型，它就是什么类型，通过不断练习，就会自然而然的理解SQL注入的各种类型。(纯属个人理解)

2.以下只是阐释具体SQL注入类型的核心/概念，不会提供具体的例子->所有的payload或注入技巧会在解题过程中集中体现（知道是什么,过过脑）

1.基于输入类型的分类

1.字符型

核心特征：输入数据需用 ‘（单引号）或 “（双引号）包裹，通过闭合引号破坏 SQL 结构。 闭合方式: ‘ “ ) ‘) “)
原理：原始 SQL 拼接用户输入时未正确转义引号，导致恶意语句介入。
防御：对字符串参数使用 预编译语句（Prepared Statements） 或 参数化查询，避免直接拼接。

2.数字型

核心特征：输入数据为纯数字，无需引号闭合，直接通过逻辑运算符破坏条件。
原理： ?id= 逻辑语句框架下+mysql语句
防御：对数字参数校验类型，确保输入为纯数字；同样使用预编译语句。

2.基于执行结果的分类

1.”信息”有回显

核心特征：SQL 执行结果直接返回给前端，可通过页面内容快速判断注入点。
常见场景：
1.页面显示查询结果（如用户信息、商品列表）。
2.数据库错误信息直接暴露（如 MySQL 的 You have an error in your SQL syntax）//及报错注入
防御：避免在前端暴露详细错误信息，统一返回通用错误提示。

2.”信息”无回显（盲注）

核心特征：SQL 执行结果不回显，需通过 间接方式 判断攻击是否成功。
分类：

1.bool型盲注：通过构造布尔条件（如 IF(条件, 1, 0)），根据页面是否正常响应判断结果
2.时间盲注：通过 SLEEP() 等函数制造延迟，根据响应时间判断条件是否成立

攻击难点：

1.需编写脚本逐字符爆破数据（如数据库名、表名、列名）。
2.防御方可能限制请求频率，增加攻击难度。

3.基于攻击手法的分类

1. 联合查询注入（Union-Based Injection）

原理：利用 UNION/UNION ALL 合并多个查询结果，获取额外数据。
注意查询前后字段数一致(要符合mysql语法):,测试语句 order by 数字
防御：限制 UNION 等危险关键字的使用，或通过预编译语句过滤

2. 报错注入（Error-Based Injection)

原理：利用数据库错误信息回显，获取敏感数据（如通过 UPDATEXML、ERROR 函数)
解释:输入的语句满足mysql语法,但不满足所使用函数的语法->报错回显
防御：关闭数据库错误信息的前端显示，使用统一的错误处理机制。

3. 堆查询注入（Stacked Queries Injection）

核心特征：通过 ; 分隔符执行多条 SQL 语句，实现增、删、改等高危操作。
前提条件：数据库连接允许执行多条语句（如 PHP 的 PDO::ATTR_EMULATE_PREPARES = true 时可能触发）。
防御关键：

禁用数据库连接的 multi_query 功能（如 PHP 的 mysqli_multi_query() 禁用）。
对用户输入严格过滤 ;、– 等符号。

4. 宽字节注入

核心特征：利用数据库字符集（如 GBK）的宽字节特性，绕过 \ 转义符。
原理：

正常情况下，' 会被转义为 '（安全）。
宽字节字符（如 %df）与 \（%5c）组合时，会被解析为一个字符（如 %df%5c → 縗），使 ' 变为 %df’，从而绕过转义。

防御关键：
数据库、表、字段统一使用 utf8mb4 字符集（不支持宽字节攻击）。
在转义前设置正确的字符集（如 PHP 的 set_charset(‘utf8mb4’)）。

4.特殊场景注入类型

1. JSON 注入

场景：参数以 JSON 格式传递时，未正确转义引号或特殊符号。

1.开启gpc或addslashes
2.数据库字符集转为GBK

防御关键：

对 JSON 数据进行严格校验（如使用 JSON 解析函数，拒绝非法格式）。
对 JSON 中的字符串字段进行转义（如将 ‘ 转为 '）。

2. HTTP 头注入

场景：攻击点位于 HTTP 请求头（如 Cookie、User-Agent、Referer）。
防御关键：

对所有用户可控的输入（包括请求头）进行统一过滤和参数化处理。
避免在 SQL 中使用请求头字段，如需使用，必须通过预编译语句传递。

3. ORM 注入

场景：使用 ORM 框架时，错误拼接查询条件（未使用参数化）。
防御关键：

遵循 ORM 框架的安全规范，使用官方推荐的参数化查询方式。
避免使用 extra()、raw() 等拼接原始 SQL 的方法，必须使用时严格过滤输入。

4. XML/RPC 注入

场景：通过 XML 格式的 RPC 请求传递参数，未转义特殊字符（如 <、&）。
防御关键：

使用 XML 解析库的安全模式（如禁用 DTD 解析）。
对 XML 中的数据进行 HTML 实体转义（如 ‘ 转为 '）。

提示使用url栏写入文件（写马）

mysql中的file系列函数

  1. 核心函数：

    • into dumpfile
      以二进制格式写入文件，适合写入二进制数据（如 WebShell），不会转义特殊字符，写入内容精确。
    • into outfile
      以文本格式写入文件，会对特殊字符（如换行符）进行转义，适合导出文本数据。
    • load_file
      读取文件内容并返回为字符串，用于读取服务器上的敏感文件（如配置文件、密码文件）。

  2. 关键限制：secure_file_priv
    MySQL 通过此系统变量控制文件导入/导出权限：

    • 值为空：允许读写任意路径（需 MySQL 进程有权限）。
    • 值为目录路径（如 /var/tmp）：仅允许在该目录下操作。
    • 值为 NULL：完全禁止导入导出功能。

  3. 写入 WebShell（写文件）

  • into outfile 与 into dumpdfile 的区别：

    函数 数据格式 特殊字符处理 适用场景
    into outfile 文本格式 自动转义（如 \n） 导出文本数据
    into dumpfile 二进制格式 无转义 写入 WebShell/二进制文件

  • 为什么优先用 into dumpfile 写 WebShell？
    避免转义破坏代码结构（如 PHP 标签 <? 被转义会导致失效）。

  1. 读取敏感文件（读文件）
  • LOAD_FILE() 要点：
    • 返回文件内容作为字符串。
    • 需绝对路径，且 MySQL 进程有读权限。
    • 常见目标文件：
      • Linux: /etc/passwd, /etc/shadow, 配置文件（如 /var/www/config.php）
      • Windows: C:\Windows\System32\drivers\etc\hosts, C:\xampp\htdocs\config.ini

  1. secure_file_priv 绕过：

    • 若值为目录路径（如 /tmp），尝试在此目录写入/读取文件。
    • 利用 MySQL 日志文件、慢查询日志等替代写入路径（需条件）。
    • 使用 NULL 字节截断（部分旧版本有效）：'C:/a.php\\0.txt'。

  2. 防御建议：

    • 配置 MySQL：secure_file_priv = NULL（禁止文件操作）。
    • 最小权限原则：MySQL 进程权限降权，禁止访问 web 目录。
    • 修复 SQL 注入：使用参数化查询（Prepared Statements）。

注入者角度利用此漏洞

1. 什么时候要用？

  • 读取服务器文件内容：当需要将数据库服务器上的文件内容（如文本、日志、配置等）直接导入到数据库表中或用于SQL逻辑处理时。
  • 数据迁移/备份：配合 SELECT ... INTO OUTFILE 导出数据后，再用 LOAD_FILE() 快速验证或读取导出的文件。
  • 动态加载内容：例如在存储过程中读取外部配置文件。

2. 什么时候可用？

需同时满足以下条件：

  • 用户权限：
    • 用户必须拥有 FILE 权限（全局权限）。
    • 授权命令：
  • 文件路径限制：
    • 文件必须位于 secure_file_priv 指定的目录内（MySQL 5.5+默认启用）。
    • 查看该目录：
    • 若值为 NULL，则禁止所有文件操作；若为空字符串，允许任何路径（不安全）。
  • 文件权限：
    • MySQL 服务进程用户（如 mysql）需有该文件的 读权限。
  • 文件要求：
    • 文件必须存在且为文本文件（二进制文件可能显示乱码）。
    • 文件大小不超过 max_allowed_packet（默认 4MB）。

  • 返回 NULL 的原因：
    • 文件路径错误（未在 secure_file_priv 目录下）。
    • 权限不足（用户无 FILE 权限或系统文件权限不足）。
    • 文件不存在或文件名大小写不匹配。
