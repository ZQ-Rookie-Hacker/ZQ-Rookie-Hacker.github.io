奇怪的认知:
1.原子化认证

第一步：输入用户名密码。
服务器校验：仅验证正确性，绝不下发能访问资源的 Session ID。
第二步：输入验证码。
服务器校验：确认验证码正确后，才生成一个全新的、高权限的 Session ID 并返回给浏览器

2.阶段化认证

第一步（密码）：服务器确认 “你是谁”。为了让你跳转到第二步页面 /login2，服务器必须给你一个 “临时入场券”（Session）。
关键点：这个 Session 处于 “半登录” 状态（只证明了你有账号密码，但还没证明你是本人）。
第二步（2FA）：验证 “你是本人”。验证通过后，系统可能只是更新这个 Session 的权限，而不是直接替换它。

疑惑：明明原子认证更加安全，为什么很多大型的网站依然使用阶段化认证

1.状态保持：如果你在第一步输入了正确的账号密码，然后去接水喝，回来发现验证码输入框消失了，或者页面跳走了，你会很烦躁。系统需要保留你 “已通过密码” 的状态。
2.多设备支持：有时候 2FA 是发到手机上的，用户可能需要几秒到几十秒才能看到。如果不先给个临时 Session，超时后用户就得重新输密码，体验极差。
3.资源隔离：那个 “半 Session” 通常只能访问 /login2 页面和一些静态资源，访问 /my-account 会被拦截

这个环境的漏洞是 资源隔离 存在漏洞，即 系统未对仅完成密码验证、未通过 2FA 的半登录态 Session 实施严格的访问控制，导致该 Session 可直接越权访问需完整 2FA 认证的受保护资源（如/my-account），从而绕过双因素认证流程

session不变的意义

Session 从头到尾不变，它的核心意义只有一个：
直接证明 —— 这套密码重置逻辑，完全不依赖「当前会话身份」做校验，这就是漏洞能成立的根本原因

  1. 说明：密码重置和「当前登录身份」彻底脱钩

    正常安全设计里，密码重置会做绑定：
    重置 token
    当前 Session
    目标用户名
    这三者必须一一对应，才能改密。
    而这里 Session 不变，说明：后端根本没检查 “这个 Session 是谁的”你哪怕是未登录访客，照样能重置 carlos 密码。
    2. 说明：上述所利用操作完全不需要 “伪造会话”

    正是因为 Session 不变、也不校验，你只需要抓包改个 username=carlos，就能直接覆盖目标用户。
    如果 Session 会变、会绑定身份，你这种改包操作直接失效。
    3. 暴露了最关键的逻辑缺陷

    Session 不变这件事本身，就是在告诉你：这个系统的密码重置权限，只由两个东西控制：
    token + 提交的用户名
    和当前会话无关，和你是谁无关。
    这也是典型的业务逻辑越权漏洞。

  • 对于所掌握的信息的再次解读
    • 服务器不会校验输入邮箱与所要修改密码的用户邮箱
      • 重置请求阶段」校验缺失：服务器仅检查「用户名存在」，完全不验证「填写的邮箱是否为该用户的注册邮箱」，允许你用自己的邮箱接收任意用户的重置token
    • 账户的密码是与其邮箱绑定的
      • 重置执行阶段」校验缺失：正常逻辑下token仅允许修改绑定邮箱的用户密码，但该环境服务器不校验token与要修改的用户名是否匹配，只要token有效，就可修改任意用户的密码

  • 开发者的逻辑

    • 锁定只针对错误密码”
      开发者认为：锁定的目的是阻止连续的错误尝试，而不是阻止账户所有者本人。如果用户输入了正确的密码，说明他是合法用户，系统应该允许他登录，并且登录成功后自动清除失败计数，恢复正常状态。

    • “用户体验优先于安全”

开发者可能担心：如果合法用户因为忘记密码或其他原因触发了锁定，却连自己的正确密码都无法登录，会造成糟糕的用户体验（例如需要等待锁定时间结束或联系客服）。因此，他们设计了一条“后门”：正确密码可以绕过锁定。

- “错误消息统一太麻烦”

开发者没有意识到锁定提示与普通错误提示的差异会泄露信息。他们可能简单地认为：锁定状态返回专门提示是为了“告诉用户账户被锁了”，而正确密码登录成功是“正常流程”，不需要额外处理。

特性 X-Forwarded-Host (XFH) X-Forwarded-For (XFF)
全称 X-Forwarded-Host X-Forwarded-For
核心作用 记录客户端原始请求的主机名/域名 记录请求经过的IP地址链，包含客户端真实IP
产生背景 代理/负载均衡器常修改Host头，XFH保留原始目标域名 代理/负载均衡器转发后，后端只能看到最后一个代理IP，XFF保留完整IP链路
典型场景 帮助后端生成正确URL、重定向和Cookie域 日志记录、访问控制、地理定位等需要真实客户端IP的场景
标准状态 非官方标准，事实上的行业规范 非官方标准，被RFC 7239部分采纳

知识点的用途是劫持用户的密码重置令牌，从而完全接管其账户。
其核心是利用中间件（Middleware）或应用对HTTP头部的信任，篡改密码重置邮件中的链接，当受害者点击该链接时，其密码重置令牌就会被发送到攻击者控制的服务器上

0x01.burpsuite- session handling rule

  • Use cookies from the session handling cookie jar
    这个动作会从 Burp 的“会话处理 Cookie 存储区”中取出所有可用的 Cookie，并自动添加到当前即将发出的请求里。通常配合宏使用：宏执行登录后，服务器返回的新 Cookie 会被存到 Cookie Jar 里，然后这个动作会把它们粘贴到后续的每一个请求中，实现会话保持。

  • Set a specific cookie or parameter value
    允许你手动指定一个固定的 Cookie 或请求参数（比如 token=abc123），并强制将当前请求中的对应值替换成你设定的值。如果你知道某个参数必须是一个固定值（而不是动态获取），就可以用这个动作硬编码进去。

  • Check session is valid
    这是在配置的“会话有效性检查”。它会分析服务器返回的响应（例如检查响应体里有没有 login 字样，或者状态码是不是 302），判断当前会话是否仍然有效。如果判断为无效，就可以触发后续动作（比如重新运行宏登录）；如果有效，则跳过后续的动作，继续发送原请求。这个动作也可以设置“每 N 个请求只检查一次”的频率。

  • Prompt for in-browser session recovery
    当 Burp 检测到会话可能失效时，它会弹出一个对话框，让你手动在浏览器中完成登录（或其他操作），然后 Burp 捕获这次浏览器交互中产生的新会话信息，并用它来更新后续请求。适用于那些自动化宏很难处理的复杂登录流程（比如验证码、OAuth 跳转等）。

  • Run a macro
    直接执行一个事先录制好的宏（一系列 HTTP 请求的序列）。宏通常用于自动登录、获取 CSRF Token、提交表单等。执行宏之后，你可以选择用宏的最终响应来更新当前请求的参数、Header 或 Cookie。

  • Run a post-request macro
    与普通宏类似，区别在于它的执行时机：在当前请求发送完成并且收到响应之后才运行宏。适合那些需要根据当前请求的响应内容来决定下一步做什么的场景，比如先提交数据，再根据返回的页面提取某个值，然后自动执行另一个请求。

  • Invoke a Burp extension
    调用一个由 Burp 扩展（如你从 BApp Store 安装的插件）提供的自定义动作。这给了你无限的可能性，因为扩展可以用 Java/Python 编写任何你需要的逻辑，比如根据请求内容动态生成签名、修改数据包等。如果内置动作不够用，就靠这个。

  • Set a specific header value
    强制为当前请求添加或修改某个 HTTP 头（例如 X-Custom-Header: myvalue）。与“Set a specific cookie or parameter value”类似，但专门针对 Header 字段。

  • Replace matching parts of the request
    使用正则表达式或字符串匹配，将当前请求的原始数据（包括请求行、Header、Body）中符合模式的部分替换成你指定的新内容。这是一个非常灵活但需要小心使用的动作，可以用来批量修改路径、参数名、甚至整个请求体格式。 可以的，最好不要太口语化

0x02. 🔑 2FA的认证形式

双因素认证的核心是使用两种不同类型的验证因子来确认身份。这主要基于以下三大类因子：

  • 你知道的（Knowledge Factor）：用户已知的信息，如密码、PIN码或安全问题的答案。
  • 你拥有的（Possession Factor）：用户实际拥有的物品，如手机（接收验证码）、USB安全密钥、银行卡等。
  • 你是什么（Inherence Factor）：用户的生物特征，如指纹、虹膜、面部识别或声纹。

以下是基于这些因子的常见 2FA 具体形式：

  • 短信验证码 (SMS OTP)
    认证因子：你知道的（密码）+ 你拥有的（手机）
    说明：登录时，网站向用户注册手机发送一次性验证码，用户输入密码和验证码完成登录。这是最常见的 2FA 形式，但安全性相对较低（易被拦截或社会工程学攻击）。

  • 身份验证器应用 (TOTP)
    认证因子：你知道的（密码）+ 你拥有的（手机）
    说明：使用 Google Authenticator、Microsoft Authenticator 等应用，基于时间生成6-8位数字的一次性密码（OTP）。比短信更安全，因为不依赖不安全的通信信道。

  • 推送通知 (Push Notification)
    认证因子：你知道的（密码）+ 你拥有的（手机）
    说明：用户登录后，已认证的设备（如手机）会收到一个登录请求推送，用户只需点击“批准”即可完成二次验证。这种方式的用户体验极佳，安全性也较高。

  • 硬件安全密钥 (Hardware Security Key)
    认证因子：你知道的（密码）+ 你拥有的（物理密钥）
    说明：物理设备，如YubiKey，通过USB或NFC连接到设备。登录时，用户需要按下密钥上的按钮来完成验证。这被认为是目前最安全的 2FA 形式之一，因为它基于公钥加密技术，能有效防御网络钓鱼和中间人攻击。

  • 生物识别 (Biometric 2FA)
    认证因子：你知道的（密码）+ 你是什么（生物特征）
    说明：利用指纹、面部识别、虹膜扫描等唯一生物特征进行验证。这种方式使用方便，但生物特征数据一旦泄露则无法更改。

  • 基于位置的认证 (Location-based 2FA)
    认证因子：你知道的（密码）+ 你拥有的（地理位置）
    说明：通过检测用户的 IP 地址或 GPS 数据来确认其地理位置是否在可信任的范围内（例如公司内部）。通常作为辅助的验证方式。

  • 备份码 (Backup Codes)
    认证因子：你拥有的（备份码）
    说明：用户在启用 2FA 时获得的一组一次性备用码，用于在无法使用主要 2FA 方法（如丢失手机）时恢复账户访问。这是一种应急恢复机制，而非日常使用的 2FA 方法。
