第一个Content-Type（请求头里的multipart/form-data）：

相当于你在快递单上标注「这是一个多件混装包裹，用【geckoformboundaryxxx】这个胶带 / 分隔条把不同物品分开」。它只负责告诉服务器「整个请求体是怎么组织、怎么拆分的」，不管里面装的具体是什么。

第二个Content-Type（请求体里的application/octet-stream）：

相当于你在包裹里的test.php文件上贴了一个标签「这是一个二进制文件（无特定格式，可兼容任意文件）」。它只负责告诉服务器「拆分后，这个具体的数据块（文件）本身是什么类型」，不管整体包裹怎么组织。
核心本质（“不简单” 的关键）

看似都是Content-Type，实则是 **“整体组织规则” 和 “局部内容属性” 的分工协作 **，缺一不可：

没有第一个：服务器收到一堆数据，不知道哪里是文件、哪里是 “提交” 按钮的字段，会直接混乱，无法拆分解析；
没有第二个（可选，但很重要）：服务器虽然能拆分出文件，但可能无法准确识别文件格式（比如是 PHP 脚本、图片还是文本），影响后续的存储或处理逻辑。

拓展：静态文件与动态文件请求

  • 静态文件请求

    • 服务器会直接把文件内容原封不动地返回给浏览器，不经过 PHP 解析器处理，也不允许接收 POST 请求（因为静态文件不需要动态交互）。
    • 蚁剑连接时会发送 POST 请求（比如传 cmd=phpinfo();），Nginx 发现你给静态文件发 POST，就会直接返回 405 Not Allowed，拒绝这个请求。

  • 动态文件请求

    • 服务器会把文件交给 PHP 解析器处理，允许接收 POST 请求（PHP 本身就是用来处理动态交互的）。
      蚁剑连接 index.php 时，POST 请求会被正常传递给 PHP，触发代码执行。

  • 猜测过程

  • 引入知识

    • 1.为什么你的原始请求包看不到 IP，但服务器知道你的 IP

      所抓取道德HTTP 应用层报文（浏览器发的），但 IP 传输在 TCP 网络底层
      发起网络连接时，路由器 / 靶场网关会自动给服务器上报：「当前直连的IP=192.168.122.15」

    • 2.X-Forwarded-For等相关请求头

      X-Forwarded-For: 用户真实IP -> 告诉后端，这个请求经过了转发，真实用户是谁
      X-Real-Ip: 用户真实IP -> 直接告诉后端，这就是真实客户端 IP
      Client-Ip: 用户真实IP 与上面的一样

  • 增加请求头

非ip形式，依然可以返回，说明XFF是可以任意修改的

  • 引入知识-模板引擎

    • 由来

      做网站时，页面有样式、颜色、排版，PHP 代码和 HTML 混写很乱,于是程序员发明了模板引擎：
      把页面样式 / 排版 写成 → 模板文件（固定不动）
      把数据（IP / 用户名 / 内容） 写成 → 变量
      比如：

      // 1. 定义数据
      $ip = “192.168.122.15”;
      // 2. 加载固定模板（模板里写：Your IP is: {{ip}}）
      // 3. 模板引擎把 $ip 填进 {{ip}}，输出页面

    工具自动把数据填进模板，生成完整网页

    • PHP 最常用的两个模板引擎

      Twig：语法{{ 任意PHP表达式 }}
      Smarty：语法 { 任意PHP表达式 }
      即算术、变量、函数、系统命令、所有合法 PHP 代码

  • 出现两次flag的原有

    模板引擎执行 system(‘cat /flag’)：

    system() 调用 Linux 命令 cat /flag
    命令输出 Flag 内容 → 直接被 system() 打印到页面

    system(‘cat /flag’) 执行完后，会返回 Flag 内容本身（命令输出的最后一行）
    模板引擎的 {{ ... }} 会把这个返回值再次渲染到页面

  • 可以换用不会自动打印的函数，比如 exec() 或 shell_exec()，它们只会返回结果，不会直接输出：

    {{shell_exec('cat /flag')}}

Redis

  • 1.Redis定义

Redis 是一款运行在服务器内存中的高性能键值对数据库（可以简单理解为「内存版的简易数据库」），和MySQL这类磁盘数据库不同，它读写速度极快，服务器端经常用它做缓存、临时数据存储。

在CTF Web题中，内网服务器几乎都会搭Redis（默认端口6379），是SSRF漏洞最常攻击的内网服务，没有之一。

  • 2.CTF 做题必记的 Redis 核心特性
    这些特性直接决定了它为什么会被SSRF攻击，也是你这道题的关键后续考点：

    • (1) 默认端口固定：6379（记死！后续要通过SSRF访问这个端口）
    • (2) 默认无密码：安装后默认不需要密码就能连接，题目里特意在hint.php藏了密码root，是出题人加的限制，需要先验证密码才能操作
    • (3)支持远程执行命令：连接成功后，可以执行Redis自身的配置命令，能往服务器磁盘写入文件（这是CTF里利用Redis的核心！）
    • (4)轻量易访问：通过SSRF的gopher://协议就能直接向Redis端口发送命令，无需复杂交互

  • 3.一般利用过程

  • (1)Redis：服务器内网常见的内存数据库，默认端口6379，CTF中SSRF的核心攻击目标；

  • (2)密码root：这道题的Redis加了密码验证，必须用这个密码才能连接操作；

  • (3)利用目的：连接Redis后，往网站根目录写入PHP Webshell，实现远程代码执行，拿到最终flag。
