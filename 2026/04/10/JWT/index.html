0x.00.JWT基础知识

  1. 🧩 JWT 的核心结构

一个 JWT 由三部分组成，用 . 分隔：

部分 名称 说明
第一段 Header（头部） 包含令牌类型（typ: "JWT"）和签名算法（如 HS256、RS256）。Base64Url 编码。
第二段 Payload（载荷） 包含声明（claims），如主题声明（sub）、过期时间（exp）、签发时间（iat）等。Base64Url 编码。
第三段 Signature（签名） 对前两部分进行签名，防止篡改。

  1. 注册声明

    • 官方声明
      • iss (issuer)：签发方
      • iat (issued at)：签发时间
      • sub (subject)：主题（一般放用户唯一标识）
      • aud (audience)：受众
      • exp (expiration time)：过期时间
      • nbf (not before)：在此时间之前不可用
      • jti (JWT ID)：唯一编号，防重放
    • 公共声明（Public Claims）:由 IANA 统一登记维护，行业通用、无冲突风险，不属于官方注册声明但广泛使用
      • auth_time：用户实际完成身份认证的时间
      • nonce：随机值，多用于 OIDC 场景，防止重放攻击
      • azp：授权参与方，通常存储客户端 ID
      • scope：权限范围字段，定义令牌可访问的资源范围
      • client_id：调用接口的客户端唯一标识
      • sid：会话 ID，关联用户当前登录会话
      • at_hash：访问令牌哈希值，OIDC 中用于校验令牌一致性
    • 私有声明（Private Claims）:业务方自定义字段，无统一标准，JWT 渗透测试中最核心的篡改目标
      • role/roles：用户角色，常见值 user/admin/root
      • username：登录用户名
      • userId/uid/id：用户唯一标识 ID
      • admin：布尔型标识，true 代表管理员权限
      • permissions：权限数组，存储具体操作权限
      • email：用户邮箱
      • group：用户所属分组
      • is_admin：管理员权限标识

  2. ✅ JWT 的典型使用流程

    • 登录：客户端发送用户名/密码 → 服务器验证成功 → 生成 JWT 并返回。
    • 存储：客户端通常将 JWT 保存在 localStorage、sessionStorage 或 HttpOnly Cookie 中。
    • 请求：客户端在后续请求的 Authorization 头或是Cookie中携带 JWT
    • 验证：服务器收到 JWT 后，验证签名、有效期、issuer 等，从载荷中提取用户身份。

  3. 📦 JWT vs Session（传统会话）

特性 JWT Session
服务端存储 无状态（不存储） 有状态（存储会话数据）
扩展性 易于水平扩展 需要共享存储（如 Redis）
吊销能力 困难（需黑名单） 简单（删除会话记录）
载荷大小 较大（包含用户信息） 很小（仅 session ID）

实现身份认证依靠的是检查 令牌的完整性（未被篡改）和真实性（由服务器签发），就会设置JWT的比对机制

对称签名算法(HS256，即 HMAC+SHA256)：同一个密钥既用于生成签名，也用于验证签名。这个密钥必须绝对保密，不能让客户端或任何第三方知道。若服务器可以使用所存处的密钥成功对应客户端发来的jwt进行解密，则通过认证。

非对称签名算法(RS256，即 RSA+SHA256)：服务端用私钥（Private Key） 生成签名，客户端或资源服务器用对应的公钥（Public Key） 验证签名。私钥只有服务端持有，公钥可以公开，则核心是私钥。 若服务器可以使用所存处的公钥成功对应客户端发来的jwt进行解密，则通过认证

注意： jwt 的生成过程都是 Header 与 Payload 分别 Base64URL 随后将其拼接在一起，然后进行 hash，再对hash后的结果进行签名得到 signature，然后再对signature进行 Base64URL。最后三部分拼接在一起，即：Base64Header.Base64Payload.Base64Signature

说到底，无论是那种加密算法服务端的密钥是万万不能泄露的

自然会有一个疑惑就是：header中是给出签名算法的，也就是说攻击者修改了header或是payload之后，知道是使用那种哈希算法对修改后的结果进行哈希。假设，攻击者得到了签名密钥，那哈希的意义何在？ 我们又知道JWT是无状态的令牌（不存令牌本身、不存 Header/Payload、不存任何哈希值、不存用户登录状态）,也就是说 不存在 原哈希值与客户端发来的现哈希值进行比对进而确定用户身份。所以重新哈希的目的或是能力不在验证用户身份，而是 验证「当前内容」和「当前签名」是配套的

还有一个概念是：哈希比对只负责防篡改，不负责认身份。 其中的防篡改防的不是 “身份被冒充”，防的是 “令牌内容被恶意乱改”

  1. JWT的存储

    • 0x01. localStorage

      • 特点：持久化存储，数据不会随浏览器关闭、设备重启丢失，仅可通过手动清除或代码删除；JavaScript 可自由读写
      • 传输方式：需前端手动读取令牌，放入请求头 Authorization: Bearer <token> 中携带
      • 漏洞关联：极易遭受 XSS 攻击窃取令牌，攻击者获取后可直接伪造用户身份
      • 适用场景：简易项目、教学 demo、CTF 渗透测试靶场

    • 0x02. sessionStorage

      • 特点：会话级存储，关闭标签页后数据立即清空；JavaScript 可自由读写
      • 传输方式：需前端手动读取令牌，放入请求头 Authorization: Bearer <token> 中携带
      • 漏洞关联：同样存在 XSS 攻击导致令牌泄露的风险
      • 适用场景：临时会话、安全性要求较低的临时业务场景

    • 0x03.普通 Cookie

      • 特点：可设置过期时间，遵循同源策略；JavaScript 可读取
      • 传输方式：浏览器会自动随 HTTP 请求发送，无需前端手动处理
      • 漏洞关联：存在 XSS 攻击窃取令牌的风险

    • 0x04.HttpOnly Cookie

      • 特点：浏览器自动存储，JavaScript 无法访问读取
      • 传输方式：浏览器自动随 HTTP 请求发送，无需前端代码处理
      • 漏洞关联：从根源避免 XSS 攻击窃取令牌，安全性更高
      • 适用场景：企业级生产环境、对身份认证安全要求较高的正式项目

    • 0x05.前端内存变量（如 Vuex/Redux/全局变量）

      • 特点：仅存储于浏览器内存中，页面刷新或跳转后数据直接丢失
      • 传输方式：前端可直接读取使用，随请求手动携带
      • 漏洞关联：无持久化泄露风险，但无法维持登录状态
      • 适用场景：不单独使用，通常配合 Cookie 实现登录状态持久化

0x05.通过 jku 头部注入实现 JWT 身份认证绕过

本实验采用基于 JWT（JSON Web Token） 的会话管理机制。服务端支持在 JWT 头部使用 jku（JSON Web Key Set URL，JSON Web 密钥集地址）参数，但存在致命安全缺陷：在根据该参数拉取验签用的公钥时，完全未校验提供的 URL 是否属于可信域名。
实验目标：伪造一个合法的 JWT，以此绕过身份认证，访问 /admin 路径下的管理员后台，最终删除用户 carlos。
可用登录凭证：用户名：wiener；密码：peter

jku（JSON Web Key Set URL）是 JWT 标准（RFC 7515）中的可选扩展头部字段，设计初衷是为分布式 / 微服务架构下的 JWT 验签提供「公钥动态发现」能力

  • 正常安全的实现逻辑：

    • 服务端（资源服务器）不硬编码公钥，而是通过预配置的可信固定 URL（如授权服务器的/.well-known/jwks.json，符合 OpenID Connect 标准），定期拉取 JWKS（JSON Web Key Set，公钥集合） 文件；
    • JWKS 文件中存储服务端的官方 RSA 公钥集合，用于 JWT 验签；
    • 验签时，服务端仅允许jku指向预配置的可信域名 / URL，严格校验域名白名单，拒绝任何外部 / 不可信 URL。

  • 设计优势：

    • 适配分布式架构：授权服务器可动态轮换密钥，无需所有资源服务器手动更新公钥；
    • 无状态认证：服务端不存储会话，仅依赖 JWT + 可信公钥完成验签；
    • 兼容 RS256 等非对称算法，完整保留哈希防篡改、密钥信任的安全机制。

0x07.JWT 算法混淆绕过

本实验使用 基于 JWT 的机制 来处理会话。虽然服务器采用了 强安全性的 RSA 非对称密钥对 进行签名和验证，但由于实现逻辑上的缺陷，导致其易受 算法混淆（Algorithm Confusion） 攻击。
实验目标：首先获取服务器的 公钥（该公钥通过标准接口对外暴露）;使用 该公钥 来签名一个 你修改过的会话令牌（JWT）；利用这个伪造的令牌访问 /admin 管理员面板，删除用户 carlos 完成实验。
可用登录凭证：用户名：wiener；密码：peter

  • 正常逻辑（RSA）

    • 密钥对：服务器持有 私钥（Private Key） 和 公钥（Public Key）。
    • 签名：服务器用 私钥 给 JWT 签名（只有服务器有，别人没有）。
    • 验签：服务器用 公钥 验证签名（所有人都能拿到公钥）。
    • 结果：只有服务器能生成合法 JWT，你无法伪造

  • 存在的漏洞逻辑

    服务器没有强制指定验签算法（比如强制写死 RS256），而是 直接从 JWT Header 的 alg 字段读取算法 去验签
    这就给了我们机会：

    1. 获取公钥：从实验室暴露的接口（通常是 /jwks.json 或 .well-known/jwks.json）拿到服务器的 公钥（Public Key）。
    2. 篡改算法：把 JWT Header 的 alg 从 RS256 改为 HS256（对称加密算法）。
    3. 关键操作：用 服务器的公钥 作为 HS256 的密钥 去签名你的 JWT。
    4. 欺骗服务器：

      服务器看到 alg: HS256，就会用 对称密钥 去验签。
      它把你传进去的 公钥字符串 直接当成了 HS256 密钥。
      因为你确实用这个密钥签过名，哈希匹配成功！
      服务器就会信任你伪造的 JWT。

总结：获取服务器公钥 → 用公钥做HS256签名 → 服务器用同一公钥验签 → 绕过

  • 如果服务器连公钥也不公开的话，这个攻击链就失效了，但是在很多真实生成场景下，公钥根本藏不住，必须公开

    1. 前端 / 客户端需要验证 JWT 合法性（比如 APP、小程序、前端页面）
    2. 微服务之间互相校验 JWT
    3. 第三方服务对接你的系统
    4. HTTPS 证书本身就自带公钥，流量抓包就能提取

  • 就算强行藏公钥，也只是「隐匿式安全」

    攻击者可以抓 TLS 证书、逆向安装包、挖源码泄露、目录遍历拿到公钥
    靠 “藏信息” 防攻击，在网安里叫Security through obscurity，行业公认是垃圾防御

一些思考：现在假设就是完全藏住了公钥，那这样对称签名算法有什么区别呢？难道’密钥’服务器存储私钥，其他提供服务的服务器存储公钥？但是这样的话所有JWT的生成都需要在’密钥’服务器产生，然后传输回去，这样符合JWT的设计初衷吗？（不清楚）那又引出了一个另外的一个问题：对称签名算法是不是不能解决上面提到的4个场景的需求？

回答：非对称签名算法 是 分布式 / 多角色场景的唯一解。集中密钥服务器 + 公钥分发 是JWT最安全最普遍的标准架构

  • 标准的 RSA 公钥，核心就两个东西：

    • n：模数（modulus）一个超级大的整数，二进制形式，是 RSA 加密 / 签名的核心参数。
    • e：公钥指数（public exponent）一般固定是 AQAB，对应十进制 65537。

  • 从服务器拿到的公钥的n值是什么？

    RSA 公钥模数 n 的「原始二进制大整数」，经过 Base64URL 编码后得到的字符串
    拆成三层说：

    原始 n：是一段二进制字节流（一堆不可见的 01 比特），是个超大整数。
    为什么要编码：二进制不能直接放在 JSON 里传输，所以用 Base64URL 把它转成可见字符串。
    看到的那串长字符：就是编码后的结果，不是原始 n 本身，只是它的 “文本包装”

0x08.通过算法混淆绕过JWT认证（无暴露密钥）

本实验室使用基于JWT的机制来处理会话。它使用一个强大的RSA密钥对来签名和验证令牌。然而，由于实现缺陷，该机制容易受到算法混淆攻击。
要解决本实验室，首先获取服务器的公钥。使用该密钥签名一个修改后的会话令牌，该令牌能让你访问/admin管理面板，然后删除用户carlos。
你可以使用以下凭证登录自己的账户：wiener:peter

这一题的漏洞原理与上一关是一样的，存在的区别：服务器未直接暴露公钥端点，需要从两个有效的 JWT 中暴力推导出 RSA 公钥的模数 n，再用该公钥作为 HMAC 密钥签名一个修改后的会话令牌，访问 /admin 并删除用户 carlos
